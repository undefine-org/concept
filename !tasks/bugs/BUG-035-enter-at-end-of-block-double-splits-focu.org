PLAN-005 W2.7.6 Slice A (post BUG-032 fix) found: pressing Enter at end of a block does the right server thing (creates new sibling) but ALSO Lexical's native ~KEY_ENTER_COMMAND~ runs, splitting the SOURCE paragraph in place. Result: source block's editor now contains ~<p>beta</p><p><br></p>~ AND a new ora-block exists below. Plus focus doesn't transfer to the new block \u2014 activeElement stays in source.

Two issues bundled:
1. /Double split/: keydown handler's ~preventDefault~ runs in bubble phase, after Lexical's beforeinput-driven split has already committed. Need to either:
   - Capture the key earlier (capture phase listener), OR
   - Register ~KEY_ENTER_COMMAND~ with Lexical at COMMAND_PRIORITY_CRITICAL and conditionally return ~true~ (handled) when ~_isAtEnd()~.
2. /Focus race/: server's ~push_event("focus_block_caret", ...)~ races the new block's BlockEditor hook mount. The new block may not yet have its hook attached when the event arrives. Options:
   - Server delays push by one tick (no).
   - Client queues unhandled focus_block_caret per block-id with a TTL; when a BlockEditor hook mounts, it consumes any pending focus for its id.
   - LiveView's ~phx-mounted~ JS hook fires on mount; have new blocks check a global pending-focus map.

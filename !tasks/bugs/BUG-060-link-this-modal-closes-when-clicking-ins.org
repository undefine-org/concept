* Intention & Purpose

The "Link this block" modal must stay open while the user interacts with its contents (kind select, note textarea, Create button). It should close only on: backdrop click, the X / Cancel buttons, Escape, or successful submit.

* Symptom (confirmed live, 2026-05-29)

=lib/concept_web/components/link_this_modal.ex=: the backdrop div has =phx-click="close_link_modal"=. The inner =<dialog>= tries to stop the click bubbling with =phx-click={JS.dispatch("click-stop")}=. But:

- =JS.dispatch/1= dispatches a *new* DOM CustomEvent named "click-stop"; it does **not** call =stopPropagation= on the originating click.
- grep confirms there is **no listener** for "click-stop" anywhere (=assets/=, =lib/=). It is a dead no-op.

So any click inside the dialog (textarea, select, Create) bubbles up to the backdrop's =phx-click= → =close_link_modal= fires → modal closes mid-interaction. Linking is effectively unusable in a real browser.

* Root cause

Misuse of =JS.dispatch= as if it were =stopPropagation=. The correct primitive is =Phoenix.LiveView.JS.exec=-free: bind the inner click to a JS command that halts propagation. In LiveView, =JS= has no public "stopPropagation" verb, so the idiom is one of:

- (A) Put =phx-click={JS.exec(...)}=? — no. Use the documented pattern: backdrop-only close via a dedicated element that fills the background, with the dialog as a SIBLING (not child) of the click-catcher, so inner clicks never bubble through the backdrop.
- (B) Add a tiny hook / =onclick="event.stopPropagation()"= is forbidden (no inline JS). Instead, attach =phx-click={JS.dispatch("click-stop")}= AND register a one-line client handler that calls =e.stopPropagation()= — but that's fragile.
- (C) Cleanest: restructure so the backdrop is a separate absolutely-positioned layer with =phx-click="close_link_modal"=, and the dialog is a following sibling at a higher z-index. Clicks on the dialog never reach the backdrop. This is the same pattern the command palette uses successfully (=fixed inset-0 bg-black/40= backdrop as its own element, dialog in a separate =fixed inset-0 flex= layer).

Prefer (C) — matches an existing working pattern in =command_palette_live.ex= and needs no JS.

* Scope

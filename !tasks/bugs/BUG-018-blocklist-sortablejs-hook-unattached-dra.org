* Intention & Purpose

Dragging the =<ora-block-handle>= (the ⋮⋮ next to each block on hover) reorders blocks. The drop persists via server event ~reorder_block~ which computes the new fractional index and updates the block. PubSub broadcasts the change to siblings, who re-stream.

* Context & Narrative

=block_list.js= already wires SortableJS:

#+begin_src js
Sortable.create(this.el, {
  handle: ".ora-block-handle",
  animation: 150,
  onEnd: (evt) => {
    this.pushEvent("reorder_block", {
      block_id: evt.item.dataset.blockId,
      new_index: evt.newIndex,
      old_index: evt.oldIndex,
    });
  },
});
#+end_src

(verified in audit; check for current event-name match before assuming).

Server: no ~reorder_block~ handler exists yet in =page_editor_live.ex=. Need to add one that:
1. Fetches sibling blocks ordered by ~position~ within scope of the same parent.
2. Computes the new ~position~ via ~FractionalIndex.between(left, right)~ or ~after_(last)~ / ~before_(first)~ depending on drop index.
3. Updates the dragged block's ~position~.
4. PubSub broadcast follows existing ~block_updated~ flow.

The handle's ~data-block-id~ is correctly emitted from =block_render.ex=. Drop into a parent (nested column / table cell) is out of scope.

* Root cause

In Lexical 0.17, ~selection.anchor~ is a /Point/ with ~.offset~ (number) and ~.getNode()~ (Node). The Node API does NOT have ~getOffset()~. Current code:

#+begin_src js
// WRONG (assets/js/components/ora-block.js:104)
atStart = selection?.isCollapsed?.() && node && node.getOffset() === 0;
#+end_src

Correct:

#+begin_src js
atStart = selection?.isCollapsed?.() && selection?.anchor?.offset === 0;
#+end_src

For ~_isAtEnd~: compare ~selection.anchor.offset~ against ~selection.anchor.getNode().getTextContentSize()~. After programmatic focus + selection move, the offset may not yet reflect the move because ~editor.update~ flushes asynchronously. Solutions:
- (a) Await the next animation frame / microtask before reading.
- (b) Read inside an ~editor.update(() => \u2026)~ callback rather than ~editor.read~.
- (c) For the navigation flow specifically: when the server pushes ~focus_block_caret~ with ~position: "end"~, set a flag on the receiving hook that the next keydown should be treated as "at end" until the user types or moves the caret.

Choose (b) for both methods. Run the check inside ~editor.getEditorState().read(() => \u2026)~ which sees the latest committed state.

* Initial Message

PLAN-005 W3.6 Slice J found a real regression hiding behind BUG-022's green tests: in a real browser, when TAB A renames a page, TAB B (subscribed to same `lv:page-editor-<uuid>` channel) does NOT receive any update for 10+ seconds. Only a manual reload pulls the new title from the server. Same for sidebar page-list label which stays stale even in TAB A.

BUG-022's unit test "8 tests, 0 failures" was a lie. Per the agent's own note: "Ash Notifier.PubSub defers broadcasts inside Ecto sandbox transactions, so we emit the broadcast directly in test to simulate production behaviour". That simulation bypassed the actual production code path. In prod the broadcast either never fires or is on the wrong topic.

* Root-cause hypotheses
1. `page_updated` is broadcast on `workspace:<id>:pages` topic (WorkspaceLive subscribes). But the PAGE editor LV (`lv:page-editor-<uuid>`) is a NESTED live_render, NOT subscribed to the workspace topic \u2014 it subscribes to per-page block broadcasts only. So WorkspaceLive's send_update only refreshes ITS OWN nested PageHeader live_component, but the SECOND TAB is a different WorkspaceLive process that should receive the broadcast on the workspace topic and re-render.
2. The Ash Notifier.PubSub configuration may emit on a topic that no LV subscribes to in prod. Verify `page.ex pub_sub do` block and confirm the topic matches what WorkspaceLive subscribes to in mount.
3. Sandbox-only test (the test) used `Phoenix.PubSub.broadcast/3` directly, bypassing Ash. Prod path needs Ash to actually dispatch.
4. PageList component / sidebar component subscribes to a different topic from PageHeader.

* Investigation steps
1. Check `lib/concept/pages/page.ex` `pub_sub` block: what topic is `page_updated` broadcast on? Is it interpolated correctly?
2. Check `lib/concept_web/live/workspace_live.ex` `mount/3` and `handle_params/3`: what topic does it subscribe to? Does the pattern match?
3. Test in dev (not test env): start `mix phx.server`, open two tabs, watch the server log for `[debug] BROADCAST topic=...` lines on rename. Confirm whether a broadcast event is emitted at all.
4. If broadcast IS emitted: confirm WorkspaceLive's `handle_info` pattern matches the event shape.
5. If broadcast is NOT emitted: dig into Ash notifier config (maybe `publish` selector excludes the action, or the `previous` flag is wrong, or the topic interpolation references a field that's nil at notify-time).

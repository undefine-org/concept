WHAT HAPPENED???? Please find the structural root causes and explain them to me.

* Summary
=Concept.Pages.Block= lock TTL is 30 s (=@lock_ttl_seconds=) but
=PageEditorLive= never fires =refresh_lock= on a timer. Any block focus that
outlives the TTL (i.e. nearly every real edit session) sees the AshOban
=release_expired_locks= cron flip =state= back to =:unlocked= behind the
LiveView's back. =held_locks[id]= remains poisoned until the next
focus/blur, and the LV's blur path attempts a now-no-op release.

BUG-047 made =release_lock= idempotent at the resource level so this no
longer raises =NoMatchingTransition=. The remaining defect is that
*the lock contract itself is broken*: a single user can hold a stale lock
indicator while the row is unlocked and grabbable by anyone else.

* Symptom
- User A focuses Block X — DB =state= = =:locked=, holder = A.
- 30 s pass with no further interaction.
- AshOban cron releases Block X — DB =state= = =:unlocked=, holder = nil.
- =PageEditorLive= still believes A holds the lock (=held_locks[X] == true=).
- =Presence= still advertises A's =:locked_block_id=.
- User B can now =acquire_lock= on X concurrently. Race on
  =update_content= — last writer wins, =RequireOwnLock= passes for both
  because each holds the row briefly.

* Root cause — concern spread across three files
The "lock" concept is not modelled as a single coherent thing; it lives in
fragments:

1. =Concept.Pages.Block= — resource-level state machine + actions.
2. =Concept.Pages.Block.Changes.RequireOwnLock= /
   =Concept.Pages.Block.Changes.SetLockMetadata= — server-side change rules.
3. =ConceptWeb.PageEditorLive= — socket-local =held_locks= map +
   =focus_block= / =blur_block= / =refresh_lock= handlers + Presence wiring.

No module owns "a lock is held by this user on this block, with this TTL,
refreshed every N seconds, surfaced to Presence". The keepalive timer was
silently omitted because nobody owned the lifecycle.

* Proposed fix — extract =Concept.Pages.BlockLock=
A single module that owns the lock concept end-to-end:

- Public API:
  =BlockLock.acquire(block_id, user, ws_id, opts)=,
  =BlockLock.refresh(block_id, user, ws_id)=,
  =BlockLock.release(block_id, user, ws_id)=,
  =BlockLock.held?(socket, block_id)=,
  =BlockLock.track(socket, block_id)= / =untrack/2= for =held_locks=.
- Owns the colocated JS hook that runs =setInterval(refresh, ttl/2 * 1000)=
  on focused blocks (=phx-hook=".LockKeepalive"=).
- Owns the LV \rightarrow Presence update on focus/blur.
- Wraps the Ash action errors so callers don't need to know about
  =NoMatchingTransition=, =:lock_held_by_other=, or
  =:not_lock_holder= idiosyncrasies \rightarrow returns =:ok |
  {:taken, holder_id} | :expired=.

* Tests (TDD)
- LV test: focus a block, advance time past TTL (use a swappable clock),
  assert refresh fired and =lock_expires_at= moved forward.
- LV test: focus + 10 s + crash the keepalive timer \rightarrow blur path
  releases cleanly; =held_locks= clears regardless of action result.
- Resource test: =BlockLock.acquire= twice as the same user behaves as
  =refresh=; second user gets =:taken=.
- Presence test: focus / blur flip =:locked_block_id= meta exactly once
  per transition, even when the keepalive fires in between.

* Acceptance
- =@lock_ttl_seconds= and the keepalive cadence both live in one module.
- =PageEditorLive= contains zero direct calls to =Pages.acquire_lock= /
  =refresh_lock= / =release_lock=.
- Long-running focus (> TTL) does not drift the DB state.
- Two browsers cannot both hold the lock simultaneously on a block whose
  cron release happened during user A's focus.

* Related

- Touches =ConceptWeb.Presence=, =Concept.Pages.Block.Changes.RequireOwnLock=.

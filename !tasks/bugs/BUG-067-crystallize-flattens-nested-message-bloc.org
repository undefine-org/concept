* Problem
clone_block/4 copies only [page_id, type, content, props, workspace_id] and
DROPS parent_block_id. Nested message blocks (bulleted list w/ children, toggle,
columns) are cloned as TOP-LEVEL page blocks → hierarchy flattened. Worse:
list_for_message sorts [parent_block_id: :asc, position: :asc]; Postgres orders
NULL last under ASC, so nil-parent (top-level) blocks sort AFTER their children
→ children cloned before parents (AssignAfterLastSibling appends in iteration
order). Moduledoc claims "render order" but nested content is de-nested AND
misordered. Flat single-level messages unaffected (why current tests pass).

* Fix
Build an old→new id map during clone so cloned children point at cloned parents
(set parent_block_id on the clone to the mapped new parent). Clone parents
before children (sort nil-parent first, or two-pass). Add a test with a nested
message block.

* Acceptance
A message containing a parent block + child blocks crystallizes to the page with
the SAME hierarchy and order; new parent/child ids are internally consistent.

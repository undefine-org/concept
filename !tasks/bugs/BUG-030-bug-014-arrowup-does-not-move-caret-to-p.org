* Hypotheses (root-cause first)

The browser chain on ArrowUp:
1. User in beta presses ArrowUp.
2. =block_editor.js= keydown listener fires.
3. Pushes ~nav_block~ with ~%{direction: "up", block_id: beta_id}~.
4. Server ~handle_event("nav_block", \u2026)~ in =page_editor_live.ex= resolves the previous sibling (alpha).
5. Server pushes ~focus_block_caret~ with ~%{block_id: alpha_id, position: "end"}~.
6. block_editor.js ~handleEvent("focus_block_caret", \u2026)~ runs on EVERY mounted block's hook instance. The ~block_id === this.el.getAttribute("block-id")~ gate routes the call to alpha's hook only.
7. Alpha's hook calls ~this.el.focusEnd()~ \u2014 which delegates to ~moveCaretToEnd(this._editor)~ + ~this._editor.focus()~.

Candidate failure points (order = likelihood):
- /A/ Server ~nav_block~ direction="up" returns wrong prev-sibling or no-op when blocks are ordered by fractional ~position~ but the resolution code iterates a list that was rebuilt after BUG-018's reorder support \u2014 a regression introduced by 25f4b65 or fad859d.
- /B/ ~focusEnd()~ races with the contentEditable focus model: ~moveCaretToEnd~ updates Lexical selection inside an ~editor.update(...)~ callback (async-flushed), and then ~this._editor.focus()~ runs against a stale DOM. For ArrowDown the new caret happens to land at offset 0 of beta's first text node (which is also the focus default), making the bug invisible.
- /C/ Beta's keydown listener doesn't actually fire on ArrowUp because the focus model puts the caret at offset 0 of beta and the browser's native ArrowUp moves selection within Lexical (not bubbling) before our listener sees the key.

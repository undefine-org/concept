WHAT HAPPENED???? Please find the structural root causes and explain them to me.

* Summary
The =:release_lock= AshStateMachine transition on =Concept.Pages.Block= declares
=from: :locked=. When the underlying =state= column has already drifted to
=:unlocked= (e.g. via the =release_expired_locks= AshOban cron, a sibling tab,
or a previous release attempt), invoking =:release_lock= raises
=AshStateMachine.Errors.NoMatchingTransition=.

* Reproduction
1. Acquire a block lock from =PageEditorLive=.
2. Externally release it (cron / direct SQL / second tab).
3. Blur the block in the original LV. The =release_if_held/2= helper still sees
   =held_locks[id] == true= and fires =Pages.release_lock/2= → server logs the
   =NoMatchingTransition= error from =update_query/4= at =lib/data_layer.ex:1833=.

Observed in production log slice 2026-05-17:
#+begin_example
[:unlocked, nil, :unlocked, [:locked], "AshStateMachine.Errors.NoMatchingTransition",
 "target", "unlocked", "field", "lock_acquired_at", "action", "release_lock",
 "old_state", :unlocked, …]
#+end_example

* Root cause
Two sources of truth for "is this block locked?":
- =Concept.Pages.Block.state= (state machine column, mutable by cron/admin)
- =PageEditorLive.assigns.held_locks= (in-memory map, mutated only on focus/blur)

The transition is closed (=from: :locked=), so any drift turns =:release_lock=
into a hard error. =release_if_held= silently swallows the error
(=_ -> socket=), keeping the poisoned =held_locks= entry alive for the rest
of the session.

* Fix (chosen path)
Make =:release_lock= idempotent at the resource level:
#+begin_src elixir
transition :release_lock, from: [:locked, :unlocked], to: :unlocked
#+end_src

The remaining changes in the action (=set_attribute(:lock_state, :unlocked)=,
clearing holder/timestamps) are already idempotent.

Also clear =held_locks[id]= on the LV side regardless of the action result
(after the resource-level fix, any non-:ok result is a real bug worth surfacing).

* Tests
- Resource test: acquire → release → release again should both succeed and
  leave =lock_state == :unlocked= with =lock_holder_id == nil=.
- Regression test: simulate the cron path (release_lock as system actor) then
  user re-release should not error.

* Acceptance
- =mix test test/concept/pages/block_lock_idempotency_test.exs= passes.
- Server log no longer contains =NoMatchingTransition= entries for
  =release_lock=.

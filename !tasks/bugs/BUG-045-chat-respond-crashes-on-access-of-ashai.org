WHY???? Root cause analysis + BUG filing proposal please

* Summary
Every streaming chat response crashes the Oban worker with =UndefinedFunctionError= on
=AshAi.ToolLoop.Result.fetch/2=. Result: partial response text appears in DB with
=complete: false=, then the action errors; user sees a stuck/incomplete message.

* Reproduction
1. Send any chat prompt in the workspace chat panel.
2. Oban worker invokes =Concept.Knowledge.Chat.Message.respond/1=.
3. Crash on first =\{:done, _\}= event.

* Stack

  (elixir) Access.get/3
  (elixir) Kernel.get_in/2
  lib/concept/knowledge/chat/message/changes/respond.ex:193
#+end_example

* Root cause
=respond.ex:193= calls =get_in(metadata, [:usage, :prompt_tokens])= on the
=\{:done, metadata\}= stream event. As of =ash_ai 0.6.1=, =AshAi.ToolLoop.stream/2= emits
=\{:done, %AshAi.ToolLoop.Result\{\}\}= — a struct that doesn't =@derive Access=. The
guard =is_map/1= admits structs, so we fall into the map clause and crash.

* Fix
1. Add explicit =%AshAi.ToolLoop.Result\{\}= clause that returns =%\{\}= (no usage data).
2. Tighten map clause with =is_map(m) and not is_struct(m)=.
3. Keep catch-all returning =%\{\}=.

* Tests
- Unit: =extract_usage_metadata/1= on =%AshAi.ToolLoop.Result\{\}= → =%\{\}=, no raise.
- Unit: =extract_usage_metadata/1= on plain map → expected shape.

* Acceptance
- =mix test= passes new regression test.
- Manual: chat in dev, message marked =complete: true=, no UndefinedFunctionError in logs.

- See [[id:BUG-usage-tokens-never-recorded]] for the latent issue of usage data
  never being populated even when the crash is fixed.

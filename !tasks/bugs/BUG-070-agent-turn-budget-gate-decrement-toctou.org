* Problem
needs_host_response gate (agent_turn_budget > 0) and the in-flight decrement
(do_stream → decrement_budget) are not in one transaction. Two messages enqueued
while budget==1 can each evaluate the gate true before either commits its
GREATEST(?-1,0) decrement → one extra host turn beyond the cap. Bounded over-run
proportional to concurrency, NOT a loop-escape (floor + per-message re-enqueue
still terminate). Errs safe.

* Fix (optional)
Decrement-and-check in one atomic step, or gate the respond worker on a
conditional decrement that returns whether a turn was granted. Only worth doing
if precise budget accounting matters.

* Acceptance
Under N concurrent host-addressed messages at budget==1, at most 1 host turn runs.

* Initial Message

Frontend audit (2026-05-16). =Concept.Accounts.Reactors.Onboarding= correctly creates a personal Workspace + owner Membership atomically post-register. But =sign_in_route= / ~confirm_route~ in =router.ex:46-77= use ash_authentication_phoenix defaults, which redirect to ~/~. =HomeLive= (=lib/concept_web/live/home_live.ex:1-67=) shows a "Get started" CTA the user has to click. New users see "Concept" landing, conclude there's nothing here, and bounce. Workspace exists in the DB but is invisible.

* Intention & Purpose

After register-confirm-magic-link, redirect the user to their primary workspace at ~/w/<slug>~, with empty-state CTAs ("Pick a page or create one") instead of a marketing landing. For returning sign-in flow, redirect to the most-recently-visited workspace (or primary if no preference).

* Context & Narrative

ash_authentication_phoenix exposes ~on_mount~ chain plus a ~live_view= hook for post-auth navigation. Idiomatic Ash pattern: implement a custom on_mount hook ~ConceptWeb.LiveUserAuth.AfterSignIn~ that, on session establishment, resolves the user's primary workspace and ~push_navigate(to: ~p"/w/#{slug}")~.

Primary workspace resolution: =Concept.Accounts.Workspace= has no ~:primary~ attribute today. Two options:
- (A) Add ~:primary?~ boolean default false; Onboarding reactor marks the personal workspace primary.
- (B) Pick first owner-membership workspace by created_at.

Choose (A) — Onboarding already runs once per user; making the bit explicit avoids reading the wrong workspace when the user joins others later.

Alternative path: instead of an on_mount hook, override ash_authentication_phoenix's ~AuthController~ success callback. Less idiomatic in LiveView world; prefer on_mount.

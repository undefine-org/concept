* Initial Message

PLAN-005 W2.6 Slice F puppeteer (2026-05-16) found three real defects in the slash menu that unit tests didn't catch:

1. /Filter "h1" returns zero matches/. =assets/js/components/ora-slash-menu.js= filter does ~label.includes(filter) || type.includes(filter)~ literally. "h1" doesn't match "Heading 1" (case + space) nor "heading_1" (underscore vs nothing).
2. /BUG-028 trigger deletion ineffective in browser/. =assets/js/hooks/slash_menu.js= ~_handleSelectItem~ calls ~root.clear() + $createTextNode(...)~ inside ~editor.update~ but the source paragraph still shows "/" after select. Likely async-flush ordering OR a Lexical-scope issue where the editor reference is stale by the time update runs.
3. /Hook _handleInput document listener doesn't fire/. Puppeteer ~type~ generates input events that reach the document (verified) but the hook's listener doesn't process them. Possibly Lexical's own input handler ~stopImmediatePropagation~'s the event before document-level capture. Or the listener is registered on ~document~ but Lexical handles at the editor root level.

* Intention & Purpose

- Filter is forgiving: case-insensitive, matches against both label and a normalized type key (`heading_1` => "h1" matches).
- Selecting an item ACTUALLY removes the slash + filter from the source block in the real browser, not just in vitest mocks.
- Slash trigger detection works regardless of how the user inputs "/" (keyboard, paste, etc.).

* Hypotheses + fixes

** Filter fix
=assets/js/components/ora-slash-menu.js= filter expression:
#+begin_src js
const f = filter.toLowerCase();
const labelNorm = label.toLowerCase();
const typeNorm = type.replace(/[_-]/g, "").toLowerCase();
const labelKey = labelNorm.replace(/\s+/g, "");  // "heading 1" -> "heading1"
return labelNorm.includes(f) || typeNorm.includes(f) || labelKey.includes(f);
#+end_src

** Trigger deletion
Move the deletion logic to a microtask after ~pushEvent~ resolves, OR ensure ~editor.update(() => \u2026)~ runs against the FOCUSED editor reference (capture via ~document.activeElement~ at select-time, not at trigger-time). Test by reading source block text /after/ a setTimeout(0) settle in the test.

** Input listener
Switch from ~document.addEventListener("input", \u2026)~ to listening on the focused ~<ora-block>~'s ~_editor~ via ~editor.registerTextContentListener~ or ~editor.registerCommand(KEY_DOWN_COMMAND, \u2026)~. This way we receive the event through Lexical's pipeline, not in spite of it.

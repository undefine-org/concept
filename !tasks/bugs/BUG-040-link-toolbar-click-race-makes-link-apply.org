* Initial Message

PLAN-005 W5.5 Slice L: clicking the Link button in `<ora-format-toolbar>.shadowRoot` does NOT show `<ora-link-editor>`. Same click steals focus from contenteditable → `selectionchange` fires with empty selection → FormatToolbar hook's `_activeEditor` is nulled → `_handleRequestLink` returns early. The Slice E dispatch-CustomEvent-directly workaround that worked in W2.7.6 ALSO fails now because `_activeEditor` is null. Link UX completely broken in real browser.

Originally filed as FUP-021 (polish). Slice L promotes it to BUG-040 — this is a P1 blocker for the link feature.

* Fix sketch
- Add `mousedown` (with `preventDefault`) to each button inside `<ora-format-toolbar>` shadow. Prevents focus loss on click.
- AND/OR in FormatToolbar hook: debounce `selectionchange` for ~200ms after any button click; ignore selection events during that window.
- AND/OR cache `_activeEditor` for ~500ms post-toolbar-show; don't null it on selectionchange unless the new active element is outside our ora-block.

Three converging fixes; pick the one most surgical (mousedown.preventDefault is usually cleanest).

Integration test that simulates the actual click → selectionchange race and asserts `<ora-link-editor>` becomes visible.

* Intention & Purpose
Restore the per-block hover affordance ("+" / "⋮⋮" handle) and the floating selection bubble (bold / italic / link / code) so users can manipulate blocks and inline-format text the way the visual design intends.

* Context & Narrative
The Lit custom elements ~ora-block-handle~ and ~ora-format-toolbar~ are defined and registered in =assets/js/components= but no HEEx template instantiates them, so they never appear in the DOM. Both elements are self-positioning Lit components: the handle is shown via Tailwind ~group-hover:opacity-100~ on the parent row, and the format toolbar listens to ~document.selectionchange~ and positions itself relative to its ~offsetParent~.

Definitions:
- =assets/js/components/ora-block-handle.js= — adds two buttons that emit bubbling ~add-below~ and ~open-menu~ custom events.
- =assets/js/components/ora-format-toolbar.js= — toggles a hidden popover when there is a non-empty range selection inside an =<ora-block>=; emits ~toggle-format~ and ~request-link~ custom events.

This bug is purely a wiring bug; the components themselves work.

* Scope
- IN: render ~<ora-block-handle>~ inside the per-block row; render exactly one ~<ora-format-toolbar>~ inside the page editor; add the CSS rules needed for the row layout and hover reveal.
- OUT: handling the new bubbling events on the server (~add-below~, ~open-menu~, ~toggle-format~, ~request-link~) — those are out of scope for this bug. Listeners may be stubbed but no =handle_event= clauses are required in this fix. Slash menu / link editor wiring is also out of scope.

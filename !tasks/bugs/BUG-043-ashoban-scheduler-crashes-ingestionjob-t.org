The cron-scheduled scheduler module
`Concept.Knowledge.IngestionJob.AshOban.Scheduler.Process` raises every
minute in production logs:

  (ash) Ash.Actions.Read.run/3
  (ash_oban) define_schedulers.ex:193 …Scheduler.Process.insert/1
  (ash_oban) define_schedulers.ex:258 …Scheduler.Process.perform/1
#+end_example

* Root cause
~IngestionJob~ is multi-tenant by ~workspace_id~ with ~global? false~.
The AshOban scheduler streams ~state == :queued~ rows without a tenant,
so the generated ~Ash.read!~ call is rejected.

The trigger does not configure ~list_tenants~. ~AshOban~ falls back to
~AshOban.Info.oban_list_tenants!/1~ which yields ~[nil]~, hitting the
~global? false~ guard.

* Impact
- No queued ~IngestionJob~ rows ever execute via the cron path
  (workers only fire when callers manually publish, e.g. on page save).
- Stack trace floods logs once per minute.
- Backfill / retry of stuck ~:queued~ rows is broken.

* Fix (TDD)
1. Add ~AshOban.ListTenants~ implementation that returns all
   ~Workspace.id~ values under a system actor.
2. Wire it into the trigger via ~list_tenants Module~.
3. Failing test: invoke
   ~Concept.Knowledge.IngestionJob.AshOban.Scheduler.Process.perform(%Oban.Job{args: %{}})~
   after enqueuing a job; assert it returns ~:ok~ and that a worker job
   is enqueued for the workspace tenant.

* Files
- ~lib/concept/knowledge/ingestion_job.ex~ — add ~list_tenants~
- ~lib/concept/knowledge/workspace_tenants.ex~ — new behaviour impl

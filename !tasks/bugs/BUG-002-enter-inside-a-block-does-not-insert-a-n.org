* Intention & Purpose
Make the editor behave like the documented contract in =assets/js/hooks/SERVER_CONTRACT.md=: pressing Enter at the end of a block creates a new empty paragraph below and moves the caret into it; ArrowUp / ArrowDown at the block boundaries move the caret to the previous / next block; Backspace at the start of an empty block deletes that block and focuses the previous one.

* Context & Narrative
Today the keyboard contract is half-wired:

- =assets/js/components/ora-block.js= dispatches the bubbling ~ora-block-enter-at-end~ / ~ora-block-arrow-up~ / ~ora-block-arrow-down~ / ~ora-block-backspace-at-start~ events when appropriate. Enter is *not* ~e.preventDefault~-ed, so Lexical also inserts a paragraph node inside the current block.
- =assets/js/hooks/block_keyboard.js= (~BlockKeyboard~ hook) is the only listener that translates those events into ~pushEvent("insert_paragraph_below" | "nav_block" | "delete_block_merge", ...)~. It is exported in =assets/js/hooks/index.js= but *no HEEx template attaches* ~phx-hook="BlockKeyboard"~ to any element, so none of those server events are ever sent. The hook also reads ~this.el.dataset.blockId~ which would not match the actual ~block-id~ attribute (Phoenix renders ~block-id="..."~, not ~data-block-id="..."~).
- =lib/concept_web/live/page_editor_live.ex= has *no* ~handle_event/3~ clause for ~insert_paragraph_below~, ~nav_block~, or ~delete_block_merge~. If those events were ever delivered, the LiveView would crash with ~FunctionClauseError~, the channel would resync, and the user would see a page reload.

Net effect: the user presses Enter, Lexical adds an internal paragraph inside the same block, the bubbling event has no listener, and visually nothing changes (which the user describes as "page reloads and nothing shows"). For ArrowUp / ArrowDown / Backspace, exactly the same: no caret movement to the neighbor.

This bug also fixes a latent duplicate-block hazard in the existing ~add_first_block~ handler: it assigns ~[block]~ locally *and* the Ash PubSub broadcast subsequently appends the same block, producing two entries with the same DOM id.

* Scope
- IN: merge keyboard listeners into the existing ~BlockEditor~ hook; preventDefault Enter / Backspace at the right moments in =ora-block.js=; add three server handlers (~insert_paragraph_below~, ~nav_block~, ~delete_block_merge~); allow ~:position~ in =Block.create_block= so a new block can be inserted between two siblings; de-dup blocks in ~block_created~ and self-broadcast handling; remove the now-redundant ~BlockKeyboard~ hook and its registration.
- OUT: rich merge of textual content on Backspace (we only delete the empty block and move focus); slash menu or block-type conversion; multi-block selection.

#+begin_src elixir
@impl true
def handle_event("insert_paragraph_below", %{"block_id" => id}, socket) do
  user = socket.assigns.current_user
  ws_id = socket.assigns.workspace.id
  page_id = socket.assigns.page_id
  blocks = socket.assigns.blocks

  case sibling_position_after(blocks, id) do
    {:ok, position} ->
      case Concept.Pages.create_block(
             page_id,
             :paragraph,
             ws_id,
             nil,
             position: position,
             actor: user,
             tenant: ws_id
           ) do
        {:ok, block} ->
          {:noreply,
           socket
           |> assign(:blocks, insert_block_sorted(blocks, block))
           |> push_event("focus_block_caret", %{block_id: block.id, position: "start"})}

        {:error, _} ->
          {:noreply, socket}
      end

    :no_source ->
      {:noreply, socket}
  end
end

@impl true
def handle_event("nav_block", %{"direction" => dir, "block_id" => id}, socket)
    when dir in ["up", "down"] do
  blocks = socket.assigns.blocks

  case neighbor(blocks, id, dir) do
    nil ->
      {:noreply, socket}

    %{id: target_id} ->
      pos = if dir == "up", do: "end", else: "start"
      {:noreply, push_event(socket, "focus_block_caret", %{block_id: target_id, position: pos})}
  end
end

@impl true
def handle_event("delete_block_merge", %{"block_id" => id}, socket) do
  user = socket.assigns.current_user
  ws_id = socket.assigns.workspace.id
  blocks = socket.assigns.blocks

  with %{id: ^id} = block <- Enum.find(blocks, &(&1.id == id)),
       %{id: prev_id} <- neighbor(blocks, id, "up"),
       {:ok, _} <- Concept.Pages.archive_block(block, actor: user, tenant: ws_id) do
    {:noreply,
     socket
     |> assign(:blocks, Enum.reject(blocks, &(&1.id == id)))
     |> push_event("focus_block_caret", %{block_id: prev_id, position: "end"})}
  else
    _ -> {:noreply, socket}
  end
end

defp sibling_position_after(blocks, id) do
  sorted = Enum.sort_by(blocks, & &1.position)

  case Enum.split_while(sorted, &(&1.id != id)) do
    {_before, [src | rest]} ->
      next_pos =
        case rest do
          [%{position: p} | _] -> p
          _ -> nil
        end

      {:ok, Concept.Pages.FractionalIndex.between(src.position, next_pos)}

    _ ->
      :no_source
  end
end

defp neighbor(blocks, id, "up") do
  sorted = Enum.sort_by(blocks, & &1.position)
  idx = Enum.find_index(sorted, &(&1.id == id))
  if idx && idx > 0, do: Enum.at(sorted, idx - 1), else: nil
end

defp neighbor(blocks, id, "down") do
  sorted = Enum.sort_by(blocks, & &1.position)
  idx = Enum.find_index(sorted, &(&1.id == id))
  if idx && idx < length(sorted) - 1, do: Enum.at(sorted, idx + 1), else: nil
end

defp insert_block_sorted(blocks, block) do
  if Enum.any?(blocks, &(&1.id == block.id)) do
    Enum.map(blocks, fn b -> if b.id == block.id, do: block, else: b end)
  else
    Enum.sort_by([block | blocks], & &1.position)
  end
end

* Edge Cases
- *Enter while lock not yet granted* — ~insert_paragraph_below~ does not require the source block lock; it only creates a new block. Existing ~save_content~ still requires lock; unchanged.
- *Network race* — multiple clients pressing Enter on the same block simultaneously: each gets a fresh fractional position via ~between/2~, so order is deterministic by lexicographic comparison; no collisions.
- *Backspace at start with non-empty content* — =ora-block.js= already requires both ~_isAtStart()~ and ~_isEmpty()~ before dispatching ~ora-block-backspace-at-start~; non-empty blocks fall through to default browser behaviour. Server is therefore only asked to delete empty blocks. No content-merge logic needed.
- *Only block on the page* — ~delete_block_merge~ is a no-op (scenario 6) so the user never ends up with zero blocks via Backspace.
- *Stale neighbor list during a fast double-Enter* — ~neighbor/3~ recomputes from current ~socket.assigns.blocks~ on each event; ~insert_block_sorted/2~ is idempotent on id, so re-inserting the same block doesn't duplicate.

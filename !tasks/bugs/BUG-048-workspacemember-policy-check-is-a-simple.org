WHAT HAPPENED???? Please find the structural root causes and explain them to me.

* Summary
=Concept.Pages.Checks.WorkspaceMember= is implemented as an
=Ash.Policy.SimpleCheck= that synchronously runs =Ash.read_first!= on
=Concept.Accounts.Membership= every time the policy is evaluated. Each
=Block= / =Page= action triggers it ≥1× (often 2× — one per policy step), so
every lock acquire/release/refresh issues an extra
=SELECT … FROM memberships … LIMIT 1= roundtrip on top of the main query.

Observed in the production log: two identical membership selects book-end each
=release_lock= and =update_content=.

* Root cause
=SimpleCheck= runs in-process per evaluation with no caching. Membership
filtering belongs in the main SQL query: it can be expressed as an
=EXISTS (SELECT 1 FROM memberships …)= predicate fused into the action's
=WHERE= clause via =Ash.Policy.FilterCheck=.

* Fix (chosen path)
Migrate =Concept.Pages.Checks.WorkspaceMember= to =Ash.Policy.FilterCheck=
and return:
#+begin_src elixir
expr(exists(workspace_memberships, user_id == ^actor(:id)))
#+end_src

To compose, add a relationship to both =Block= and =Page=:
#+begin_src elixir
has_many :workspace_memberships, Concept.Accounts.Membership,
  no_attributes?: true,
  filter: expr(workspace_id == parent(workspace_id))
#+end_src

This emits an =EXISTS= subquery that the planner can hoist into the main
=UPDATE … WHERE= rather than a separate roundtrip.

* Tests
- Regression test that authorization still succeeds for members and fails for
  non-members against both =Block= and =Page=.
- Query-count assertion via =:telemetry_test.attach_event_handlers/2= on
  =[:concept, :repo, :query]=: =release_lock= against a held block emits 0
  separate =SELECT … FROM memberships AS m0= queries (the EXISTS is part of
  the main UPDATE).

* Acceptance
- Passing tests above.
- Production log no longer shows back-to-back =SELECT … memberships= queries
  around block actions.

* Related
- =Concept.Accounts.Membership='s own policies already use the =FilterCheck=
  pattern via inline =expr/1= — this aligns the cross-domain check with that
  convention.

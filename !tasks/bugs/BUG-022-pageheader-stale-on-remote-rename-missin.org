* Initial Message

Frontend audit (2026-05-16). =WorkspaceLive.handle_info= (lines 135-165) correctly updates ~@current_page~ + ~@page_title~ on the ~page_updated~ broadcast (PLAN-002 BUG-003 fix). But =PageHeader= is a =Phoenix.LiveComponent= rendered via ~<.live_component module={...} id={...} page={@current_page} …>=. Phoenix sends the new ~page~ assign to the component only when the parent re-renders; if WorkspaceLive renders without changing the slot (e.g. same template result), the component holds the old ~page~ struct in its own state — and worse, its own ~mount/update~ callback overwrites it on initial mount but not on parent-driven update unless explicitly re-rendered.

In practice: cross-tab rename updates browser tab title (driven by ~@page_title~ in root layout) and the sidebar (driven by ~@pages~), but the =PageHeader='s ~<h1>~ still shows the old title until you refresh.

* Intention & Purpose

Cross-tab and cross-user renames update the header immediately. Same for emoji and cover color changes (set_emoji / set_cover_color flow through the same broadcast). The component is always in sync with =@current_page=.

* Context & Narrative

Two acceptable fixes:
- (A) Explicit ~Phoenix.LiveView.send_update(PageHeader, id: ..., page: new_page)~ from WorkspaceLive's ~handle_info~ after assigning ~@current_page~.
- (B) PageHeader subscribes to the page topic itself and self-updates via its own ~handle_info~ in ~update_many~ / ~update/2~.

Choose (A): simpler, single source of truth, follows Phoenix idioms. The component remains pure: parent passes the latest page, component renders it.

Additional concern: when ~@current_page~ is reassigned but the component's id stays the same (~"page-header-#{@current_page.id}"~), Phoenix's diffing should already pass new assigns; but in practice with nested live_components there are reports of stale assigns when the parent doesn't trigger a re-render. The defensive ~send_update~ is correct and cheap.

* Initial Message

Frontend audit (2026-05-16). =assets/js/hooks/global_keys.js= captures Cmd-K and pushes ~open_command_palette~. The hook is mounted only on the WorkspaceLive root element. HomeLive (=lib/concept_web/live/home_live.ex=) and the auth pages do not mount it, so Cmd-K is silent on /, /sign-in, /register. PLAN-001 FEAT-014 implied global Cmd-K.

* Intention & Purpose

Cmd-K on the landing page (when signed in) opens a minimal palette that lets the user jump straight to their primary workspace or a recently-visited workspace. Cmd-K on /sign-in does nothing (no scope) — that's fine. Mostly a consistency fix so users don't learn to expect Cmd-K and then have it fail.

* Context & Narrative

After BUG-020 lands, signed-in users won't reach HomeLive at all (they get redirected to /w). So this BUG's surface narrows: only relevant for signed-out users on /, plus the brief moment before redirect.

Decision: mount =GlobalKeys= on HomeLive's root with a stub server handler that, when signed-in, ~push_navigate(to: ~p"/w")~ on Cmd-K; when signed-out, no-op (or focus the sign-in button).

* Scope

- IN: add ~phx-hook="GlobalKeys"~ to HomeLive's root element.
- IN: handle_event ~open_command_palette~ in HomeLive — when signed-in (redundant given BUG-020) push to /w; when signed-out, no-op.
- OUT: actually rendering a palette modal on the landing page — too invasive for this rescue.

* Initial Message

implement a pixel-perfect version of Notion. Use Ash as much as possible. This plan should cover the MVP version.

* Intention & Purpose
Make collaboration visible: presence avatars in page header (chip per active user, color-coded), block-level lock indicator (subtle border + holder avatar + holder name tooltip), and seamless propagation of remote content changes (already wired in FEAT-008, polished here).

* Context & Narrative
FEAT-007 already publishes presence and FEAT-008 already subscribes to block events; this FEAT focuses on UX polish: avatar bar, lock overlays, smooth cross-tab updates without flicker.

* Scope
** In-scope
- =ConceptWeb.PresenceBar= live_component in =PageHeader= renders up to 5 avatar chips (initials or emoji) + "+N more" overflow; reads from =Phoenix.Presence.list/1=
- Each =<ora-block>= gets =data-locked-by= attribute when held by another user; CSS adds left border in user's color + tooltip with holder name
- =BlockEditor= JS hook respects =data-locked-by= → forces =setReadOnly(true)=, hides format toolbar
- Color assignment: hash =user_id= to one of 8 preset colors (consistent across sessions)
- Notification "X user(s) editing" in sidebar page row when remote edits occur (transient toast, 2s)
- All four lifecycle events drive UI: =:block_locked=, =:block_unlocked=, =:block_updated=, =:presence_diff=

** Out-of-scope
- Cursor presence (showing exact caret of other users) — requires CRDT-ish work, FUP
- Selection presence — FUP
- Notification center for asynchronous events — FUP

* Existing Patterns
- =Phoenix.Presence.list/1= returns =%{user_id => %{metas: [%{user_id, display_name, avatar_emoji, online_at, color}]}}=
- Petal Avatar / AvatarGroup components
- Ash.Notifier.PubSub events from FEAT-006 already reach PageLive

* Tests

Scenarios:
1. Two PageLive sessions for same page; both appear in presence bar; user A leaves → bar updates within 1s.
2. User A acquires lock on block X; user B's view marks block X with =data-locked-by=user_a_id= and read-only flag.
3. User A saves content on block X; user B's =<ora-block>= receives =applyRemote= and renders new content.
4. User A releases lock; user B's block X becomes editable.
5. Presence color is stable for the same user across reconnects.

* Edge Cases
- User in 6 tabs of same workspace: presence shows once (group by =user_id= in metas).
- Lock granted to user who immediately disconnects: TTL release (FEAT-007) clears it; UI updates within 30s + 10s trigger interval.
- Presence flap (network blip): 5s grace before removing avatar.
- Apply remote arriving while user holds lock: discarded with warning (should not happen due to lock contract).
- Color collision (8 colors, >8 users): tolerated; users with same color distinguished by initials.

* Intention & Purpose

Unify auth + tenancy for MCP. After this FEAT, an API key may be
*workspace-bound*; otherwise the request must supply
~mcp-workspace-id~ header. Either way, the tenant is fixed before any
action runs.

This eliminates the need for every MCP tool to take a ~workspace_id~
argument from the LLM (and the corresponding "did the LLM pick the right
workspace?" risk).

* Mechanism

Resolution precedence (will appear verbatim in AGENTS.md + docs):

1. If the API key has a ~workspace_id~ → that workspace. Headers are
   *ignored* (no override) to prevent privilege confusion.
2. Else if request includes ~mcp-workspace-id~ AND the user is a member of
   that workspace → that workspace.
3. Else → no tenant assigned. Mutation tools refuse with HTTP 400 + JSON
   error body. Workspace-agnostic reads (~list_my_workspaces~,
   ~current_user~) proceed.

** The plug

#+begin_src elixir
defmodule ConceptWeb.Plugs.MCPWorkspaceContext do
  import Plug.Conn

  def init(opts), do: opts

  def call(conn, _opts) do
    user = conn.assigns[:current_user]
    api_key = conn.assigns[:current_api_key]  # set by AshAuthentication.Strategy.ApiKey.Plug

    case resolve_tenant(user, api_key, conn) do
      {:ok, ws_id} ->
        assign(conn, :mcp_tenant, ws_id)
      {:ok, :none} ->
        assign(conn, :mcp_tenant, nil)
      {:error, reason} ->
        conn
        |> put_status(400)
        |> put_resp_content_type("application/json")
        |> send_resp(400, Jason.encode!(%{
          error: "workspace_context_invalid",
          reason: reason,
          hint: "Either bind the API key to a workspace at creation, or pass an `mcp-workspace-id` header with a workspace you are a member of."
        }))
        |> halt()
    end
  end

  # MFA referenced from AshAi.Mcp.Router opts so the tenant comes from the conn:
  def fetch_tenant(conn), do: conn.assigns[:mcp_tenant]

  defp resolve_tenant(_user, %{workspace_id: ws_id}, _conn) when not is_nil(ws_id),
    do: {:ok, ws_id}
  defp resolve_tenant(user, _api_key, conn) do
    case get_req_header(conn, "mcp-workspace-id") do
      [ws_id | _] ->
        case Concept.Accounts.get_membership(user.id, ws_id, actor: user) do
          {:ok, %{}} -> {:ok, ws_id}
          {:ok, nil} -> {:error, :not_a_member}
          {:error, _} -> {:error, :membership_check_failed}
        end
      [] ->
        {:ok, :none}
    end
  end
end
#+end_src

* Edge cases

- API key with workspace_id pointing at a workspace the user is no longer
  a member of → plug should treat this as ~:not_a_member~ and refuse;
  document as "if you leave a workspace, revoke its workspace-bound keys."
- ApiKey rotation: preserve ~workspace_id~ from the parent key.
- Workspace-agnostic tools (e.g. ~workspace_list_my_workspaces~): the plug
  assigns ~:mcp_tenant nil~ and AshAi proceeds; the action must work
  without a tenant. Verify by adding the action to the ~Workspace~
  resource with no ~require_tenant~ constraint.

* Tests

** Plug

1. ApiKey bound to workspace A → tenant = A; header ignored.
2. ApiKey unbound, valid header for workspace B (user is member) → tenant = B.
3. ApiKey unbound, header for workspace C (user not member) → 400 + reason.
4. ApiKey unbound, no header → ~:mcp_tenant nil~ (allowed for agnostic actions).

** Cross-workspace isolation (integration)

1. Create workspaces A and B, user is member of both.
2. Create ApiKey bound to A.
3. Call ~tools/call~ ~page_list_tree~ → returns only A's pages.
4. Reading ~concept://workspaces/{B_id}/tree~ → empty / 403, *never* returns B's pages.

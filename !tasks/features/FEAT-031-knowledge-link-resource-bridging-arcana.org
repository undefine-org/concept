* Initial Message

Do a critical review of Arcana's integration into Concept and how you'd make the frontend interesting to use w.r.t those features. Go wild, go free, be thorough

* Intention & Purpose
Let users explicitly assert block-to-block relationships. Each Link row mirrors into ~Arcana.Graph.Relationship~ via an idempotent after_action upsert, enriching GraphRAG with curation signal.

* Scope (IN)
- Resource ~Concept.Knowledge.Link~ with: ~workspace_id~ (tenant), ~source_block_id~, ~target_block_id~, ~kind~ (atom: ~:relates_to | :cites | :contradicts | :see_also~), ~note~ (string), ~created_by_user_id~.
- ~ash_paper_trail~ for edit history.
- after_action change ~Concept.Knowledge.Changes.MirrorToArcanaGraph~ upserts ~Arcana.Graph.Relationship~ with ~kind: "USER_#{atom |> Atom.to_string |> String.upcase}"~ (e.g. ~"USER_CONTRADICTS"~). before_destroy removes the mirror.
- Policy: create/update/destroy by workspace members; system bypass present.
- Uniqueness: ~(source_block_id, target_block_id, kind)~.

* Tests (6 scenarios)
1. Create Link with valid kind → row + Arcana.Graph.Relationship upsert exists.

3. Destroy Link removes the mirrored graph row.
4. Cross-workspace target rejected (target must be same workspace).
5. PaperTrail records create + destroy.
6. Self-loop (source == target) rejected.

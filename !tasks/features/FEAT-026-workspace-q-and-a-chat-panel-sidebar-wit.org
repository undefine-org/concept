* Initial Message

how would you integrate Arcana into this, maximally leveraging Ash to make the system as simple as possible?

* Intention & Purpose
The flagship Arcana surface: a right-side chat panel pinned to `WorkspaceLive` where users ask questions about their workspace and watch streamed answers with clickable citations. Session-persistent across page navigations within the workspace.

* Context & Narrative
`ConceptWeb.WorkspaceLive` already manages workspace + page selection, with PageEditorLive `live_render`'d into the main column. We add a new LiveComponent `ConceptWeb.WorkspaceLive.ChatPanel` mounted in a collapsible right sidebar. The component subscribes to `"knowledge:answer:<answer_id>"` topics on demand, accumulates tokens into a LiveView `stream(:messages, ...)`, and renders user/assistant turns with citation cards.

UX:
- Toggle button in the workspace top bar (icon `hero-sparkles`).
- Panel: 360px wide, slides in from right, contains chat history, input at bottom, "New chat" button.
- User turn: bubble with their question.
- Assistant turn: streamed markdown rendering (Earmark, FEAT-018), followed by a `Sources` section with citation cards `<a href="/w/.../p/<page>#block-<bid>">› Page › ... › snippet</a>`.
- Empty state: "Ask anything about your workspace" with example prompts.

State: per WorkspaceLive session, in-memory only. Persistence to DB deferred to FUP (would need a `Concept.Knowledge.Conversation` resource).

* Scope
- IN: `ChatPanel` LiveComponent; PubSub plumbing for streaming; markdown rendering of full answers; citation cards; empty state; toggle button.
- OUT: chat history persistence; multi-turn conversation memory (each question stands alone in v1); voice input.

* Existing Patterns (MUST follow)
- `<Layouts.app flash={@flash} current_scope={...}>` wraps LiveView templates per AGENTS.md.
- `<.icon name="hero-..." />` for icons.
- LiveView streams: `stream(socket, :messages, [...])` with `phx-update="stream"` on parent and unique DOM id per child.
- `phx-debounce`, `phx-mounted`, `phx-window-keydown` patterns from existing CommandPaletteLive.
- LiveComponent if scoped to `WorkspaceLive`; AGENTS.md says avoid LiveComponents unless strong reason — here the strong reason is encapsulation of subscribe/handle_info isolated from the parent LV's many concerns. **However**, LiveComponents do not handle PubSub `handle_info`. Therefore: keep panel as a LiveComponent for rendering, but `Phoenix.PubSub.subscribe/2` happens in WorkspaceLive's `handle_info` for `{:knowledge_token, ...}` and forwards via `send_update/2` to the panel. (Acceptable per AGENTS.md guidance.)

* Edge Cases
- User sends a second question before the first finishes: each gets its own answer_id; both stream concurrently. UI shows both placeholders updating; OK.
- Browser closes mid-stream: server task continues, broadcasts ignored, no leak.
- Earmark fails on weird input: rescue and render plain text.
- Workspace switch while panel open: panel resets; active subscriptions cancelled (`Phoenix.PubSub.unsubscribe`).
- Long answers (>10k chars): rendering hot; debounce token append at 50ms intervals (use `Process.send_after`).
- Empty workspace: still allow asking; will return "I don't know" — UI renders gracefully with no sources.

* Context (read in full before editing)

** The principle being enforced

After this plan lands, an Ash action with a ~description~ is automatically
projected as an MCP tool (via ~Concept.AutoTools~, FEAT-052). Therefore:

- An action WITHOUT a description is *invisible* to MCP — that may be
  intentional (internal/system actions) or a bug (forgotten public verb).
- An action WITH a description is *visible* — and its description becomes
  the LLM-readable tool documentation.

Your job: write the description for every action a human can trigger.

** Stack assumptions

- Ash 3.24.6, AshAi 0.6.1, Spark 2.7.0. ~description~ is a standard Ash
  DSL option on every action type (~create~, ~read~, ~update~, ~destroy~,
  ~action~). Per-argument description is set via the third option of
  ~argument~ (e.g. ~argument :foo, :string, description: "..."~).
- Domains in this plan use ~Concept.AutoTools~ (FEAT-053 wires it).

** Description style guide (apply uniformly)

- *One sentence, < 110 chars where possible.*
- *Imperative voice* for mutations ("Rename a page.") — *declarative* for reads ("List a workspace's pages as a hierarchical tree.").
- *Domain language*, no implementation terms ("position" not "lexorank index"). Position can be described per-arg.
- *Start with a verb for mutations*, or a noun phrase for reads.
- *Mention the unit it operates on* if not obvious from the action name
  (e.g. ~set_icon~ → "Set a page's icon emoji.").
- *No marketing fluff*, no exclamation marks, no emoji in descriptions.

Per-argument descriptions follow the same rules, focused on what the
argument means and constrains:

#+begin_src elixir
argument :workspace_id, :uuid,
  allow_nil?: false,
  description: "Workspace to create the page in. Must be a workspace the actor is a member of."

argument :position, :string,
  allow_nil?: true,
  description: "Lexorank fractional index. Pass nil to append after the last sibling."
#+end_src

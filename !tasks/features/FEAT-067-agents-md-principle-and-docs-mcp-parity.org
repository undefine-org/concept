#+begin_src markdown
## Principle: MCP parity by construction

> Every Ash action with a `description` is a feature.
> The LiveView is one projection of that feature; the MCP tool is another.
> If a human can do it in Concept, an LLM can do it via MCP —
> automatically, with the same policies, on the same data.

Practical rules (enforced by Credo + CI):

1. Every public Ash action has a one-line `description` and a description
   on every public `argument`. Undescribed = unexposed.
2. LiveView `handle_event/3` callbacks call domain code-interface fns
   only. No `Ash.Query`, `Ash.Changeset.for_*`, `Ecto.Query`, or
   `Concept.Repo`. Enforced by `Concept.Credo.Check.LiveViewPurity`.
3. Cross-resource workflows live in Reactors, wrapped as a single
   action.
4. Block types declare their LV+JS+Ash+MCP surface in one place via
   `BlockType.{Static,Interactive}` macros. See `docs/blocks/ADDING_A_BLOCK.md`.
5. Opt-out of MCP exposure is explicit: `meta expose: [mcp: false]` on
   the action.
6. Workspace tenancy: ApiKey may be workspace-bound; otherwise the
   `mcp-workspace-id` HTTP header supplies it. Bound key beats header.
   Mutation tools refuse without a tenant.

This contract is enforced by `test/integration/mcp_parity_test.exs` and
documented in `docs/mcp_parity.md`. The current MCP surface is at
`docs/mcp_surface.md` (auto-generated by `mix concept.docs.mcp_surface`).
#+end_src

* Content — docs/mcp_parity.md (outline)

Structure (~ 300-400 lines, ExDoc-compatible Markdown):

1. **Principle** — the AGENTS.md text expanded with a one-paragraph
   "why this matters" framing (CQRS lens; build for the LLM = build the
   product, not a separate workstream).
2. **Architecture** — mermaid diagram showing Action ↔ {LiveView, MCP,
   AshJsonApi-future, AshAdmin}. One paragraph per arrow.
3. **The action is the unit** — what makes an action "MCP-eligible";
   how AutoTools synthesizes the tool; what happens when AshAi sees it.
4. **Description style guide** — the same guide used by the W1 FEATs.
   Examples of good vs bad descriptions.
5. **Per-argument descriptions** — why they matter for LLMs (JSON schema
   doc generation); examples.
6. **MCP resources vs tools** — when to use each:
   - Tools = verbs (mutations + computed reads with named params).
   - Resources = nouns (canonical addressable entities the LLM browses).
   - Block UI = visual surface (MCP Apps).
7. **Workspace tenancy & auth** — the precedence ladder, exact behavior
   table, ApiKey workspace-binding flow, header fallback, refusal
   semantics.
8. **Block types** — how PLAN-006's `BlockType.{Static,Interactive}`
   macros extend to MCP: one declaration, four projections.
9. **Anti-patterns** — red/green code samples:
   - LiveView with inline `Ash.Query` → bad; refactor to a read action.
   - Manual `tool ...` duplicating an action with description → bad;
     remove the manual.
   - Per-block-type custom MCP wiring → bad; use the macros.
10. **Testing parity** — how `test/integration/mcp_parity_test.exs`
    enforces the contract; what to add when introducing a new mutation.
11. **Auto-generated surface** — links to `docs/mcp_surface.md`;
    explains its lifecycle (regenerated on every `mix concept.docs`;
    CI fails on drift).
12. **Future** — AshJsonApi/AshGraphql layering on the same actions;
    real-time MCP subscriptions; FUP cross-references (FUP-007
    intent bar; FUP-004 per-workspace keys).

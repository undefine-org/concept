* Intention & Purpose
Citations from the chat panel and AI block must scroll to the cited block on click. We add stable `id="block-<uuid>"` anchors to every rendered block and a colocated JS hook that, on mount and on `phx:page-loading-stop`, scrolls the URL hash target into view with a brief highlight animation.

* Context & Narrative
`ConceptWeb.BlockRender` (lib/concept_web/components/block_render.ex) is the single function component dispatching to per-type templates. Each text block already gets `id="b-#{@block.id}"` on the `<ora-block>` web component. We add a SECOND outer wrapper `<div id="block-<uuid>">` so citations remain stable even if the inner editor element changes. Static (non-text) blocks need the same treatment.

`PageEditorLive` is rendered via `live_render` from `WorkspaceLive`. Adding a colocated `:type={Phoenix.LiveView.ColocatedHook}` script registered with the BlockEditor target, OR registering a new external hook `PageScroll`, is the choice. Per AGENTS.md, colocated hooks belong in HEEx and are auto-bundled — choose colocated.

* Scope
- IN: outer `<div id="block-<uuid>">` wrapper around every block (text + static); CSS class for fade-highlight animation; colocated `.PageScroll` hook on PageEditorLive root reading `location.hash` on mount and on Phoenix navigation; smooth scrollIntoView; 1.5s yellow highlight class toggled on the matched element.
- OUT: link previews/popovers; collaborative cursor of "currently viewed block".

** ITEM PageScroll colocated hook
:PROPERTIES:
:CUSTOM_ID: FEAT-025-page-editor-block-deep-link-anchors-and::hook
:DEPENDS: FEAT-025-page-editor-block-deep-link-anchors-and::wrapper
:END:
- File: `lib/concept_web/live/page_editor_live.ex`.
- Locate root `<div id="page-editor-root" phx-hook=".PageScroll" ...>`.
- Add at the end of the template:
  ```heex
  <script :type={Phoenix.LiveView.ColocatedHook} name=".PageScroll">
    export default {
      mounted() { this.handleHash(); },
      updated() { this.handleHash(); },
      handleHash() {
        const hash = window.location.hash;
        if (!hash || !hash.startsWith("#block-")) return;
        const el = document.getElementById(hash.slice(1));
        if (!el) return;
        el.scrollIntoView({ behavior: "smooth", block: "center" });
        el.classList.add("block-highlight");
        setTimeout(() => el.classList.remove("block-highlight"), 1500);
      }
    }
  </script>
  ```
- Listen to `phx:navigate` to re-run handleHash after `push_patch` navigations.

* Edge Cases
- Hash points to a block that no longer exists: `getElementById` returns null; hook silently no-ops.
- LV reconnect after disconnect: `mounted()` re-runs and rescrolls — fine.
- User scrolls manually before the hook fires: `scrollIntoView({block:"center"})` is mildly disruptive; debounce: only scroll if `Math.abs(el.getBoundingClientRect().top - window.innerHeight/2) > 200`.
- Block is inside a collapsed `toggle` container: hook can't expand it. Document as known limitation in v1; FUP to expand ancestor toggles.
- Multiple blocks selected (in editor): hook only highlights the URL-target one.

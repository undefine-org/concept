* Context (read in full)

** Principle

A described public action = an MCP tool. Auth-internal actions
(~register_with_password~, ~sign_in_with_password~, ~confirm~, etc. —
generated by AshAuthentication) are NOT to be described. They are
mediated by browser auth flows, not by LLMs.

** Crucial caveat — most Accounts actions stay undescribed

Most actions on these resources are AshAuthentication-internal. Only the
following user-visible reads + the workspace-switcher set should be
exposed.

* Files

- ~lib/concept/accounts/user.ex~ — describe ONLY reads safe for the actor
  to introspect themselves; leave auth strategies alone.
- ~lib/concept/accounts/workspace.ex~ — list and read workspaces the user
  belongs to.
- ~lib/concept/accounts/membership.ex~ — list memberships.
- ~lib/concept/accounts/api_key.ex~ — list/destroy own keys (do not
  describe ~:create~ here — it's covered by FEAT-058 with a workspace
  scoping rework).

DO NOT edit ~lib/concept/accounts.ex~ (the domain file).

* Anti-patterns to avoid

- *Do not* describe AshAuthentication strategy actions
  (register/sign_in/confirm/reset). Doing so exposes them to MCP, which
  is a security regression — those flows require browser context (CSRF,
  session, redirects).
- *Do not* invent new actions in this FEAT (e.g. ~:list_my_workspaces~).
  If a needed action doesn't exist, add a TODO and defer to FEAT-058.

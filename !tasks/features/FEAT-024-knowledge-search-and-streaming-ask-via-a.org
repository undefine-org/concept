* Intention & Purpose
Implement the `:search` and `:ask` generic actions on `Concept.Knowledge`. `:search` returns ranked chunks for the command palette. `:ask` runs the Arcana Pipeline (rewrite → fusion search → answer streaming) and emits tokens via Phoenix.PubSub for any subscribed LiveView.

* Context & Narrative
Per research, `Arcana.Pipeline` accepts `collection: name` runtime opt and `Arcana.Pipeline.answer(ctx, stream: true)` returns `{:ok, stream}`. We do not control the chat model token-level — we simply iterate the stream and broadcast.

The PubSub topic convention: `"knowledge:answer:" <> answer_id` where answer_id is a per-question UUID handed back to the LiveView. The LiveView subscribes BEFORE invoking `:ask`. The action runs the Pipeline INSIDE a `Task` so it does not block the caller; on each token broadcast `{:knowledge_token, answer_id, token}`; on completion broadcast `{:knowledge_done, answer_id, %{sources: [...], full_answer: text, model: m}}`.

Citations: each retrieved chunk has `metadata.block_id` and `metadata.page_id`. We pass them through to the `:done` payload so LiveView renders citation cards.

** ITEM Concept.Knowledge.Ask (streaming task)
:PROPERTIES:
:CUSTOM_ID: FEAT-024-knowledge-search-and-streaming-ask-via-a::ask-impl
:DEPENDS: FEAT-024-knowledge-search-and-streaming-ask-via-a::search-impl

- File: `lib/concept/knowledge/ask.ex`.
  ```elixir
  def ask(question, workspace_id, opts) do
    answer_id = Ash.UUID.generate()
    topic = "knowledge:answer:" <> answer_id
    Task.Supervisor.start_child(Concept.TaskSupervisor, fn ->
      run_pipeline(answer_id, topic, question, workspace_id, opts)
    end)
    {:ok, %{answer_id: answer_id, topic: topic}}
  end

  defp run_pipeline(answer_id, topic, question, ws_id, opts) do
    name = Concept.Knowledge.Config.collection_for(ws_id)

    :telemetry.span([:concept, :knowledge, :ask], %{answer_id: answer_id}, fn ->
      try do
        ctx =
          Arcana.Pipeline.new(question, repo: Concept.Repo, collections: [name])
          |> Arcana.Pipeline.rewrite()
          |> Arcana.Pipeline.search()
          |> Arcana.Pipeline.answer(stream: true,
              llm: {Concept.Knowledge.Config.llm_model(),
                    api_key: Concept.Knowledge.Config.api_key()})

        case ctx.answer do
          {:ok, stream} ->
            full =
              stream
              |> Stream.map(fn token ->
                Phoenix.PubSub.broadcast(Concept.PubSub, topic,
                  {:knowledge_token, answer_id, token})
                token
              end)
              |> Enum.join()

            sources =
              ctx.chunks
              |> Enum.map(fn c -> %{
                  block_id: c.metadata["block_id"],
                  page_id: c.metadata["page_id"],
                  breadcrumbs: c.metadata["breadcrumbs"],
                  snippet: String.slice(c.text, 0, 240),
                  score: c.score
                } end)

            Phoenix.PubSub.broadcast(Concept.PubSub, topic,
              {:knowledge_done, answer_id,
                %{answer: full, sources: sources, model: Concept.Knowledge.Config.llm_model()}})
            {:ok, %{measurements: %{}}}

          {:error, reason} ->
            Phoenix.PubSub.broadcast(Concept.PubSub, topic,
              {:knowledge_error, answer_id, reason})
            {{:error, reason}, %{}}
        end
      rescue
        e ->
          Phoenix.PubSub.broadcast(Concept.PubSub, topic,
            {:knowledge_error, answer_id, Exception.message(e)})
          reraise e, __STACKTRACE__
      end
    end)
  end
  ```

* Edge Cases
- LiveView disconnects mid-stream: Task continues to broadcast (no consumer, harmless). On reconnect, LiveView re-subscribes BUT cannot replay; document this — partial answers are lost on disconnect. FUP: add a buffered ETS keyed by answer_id with TTL to allow replay.
- LLM rate limit: Pipeline returns `{:error, :rate_limited}`; we broadcast `:knowledge_error`.
- Empty workspace: search returns `[]`; Pipeline.answer with no chunks → uses fallback prompt → "I don't know based on this workspace."
- Question contains PII / sensitive content: out of scope for v1; no redaction.
- Caller passes `reply_to: pid`: optional; if present, also `send/2` the same messages directly to the pid (useful for tests).

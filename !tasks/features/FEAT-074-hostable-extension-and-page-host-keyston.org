* Goal
Validate the ~Concept.Hostable~ keystone against Spark reality with the smallest
slice. If ~use Concept.Hostable~ on Page is as clean as ~use BlockType.Interactive~,
the whole design holds; if not, reconsider before any further wave.

* Scope (3 files + 1 migration)
1. ~Concept.Hostable~ Spark extension, two halves mirroring repo idioms:
   - mixin (~__using__~): ~@behaviour~ + ~subgraph_scope/1~ callback with
     ~defoverridable~ (like ~BlockType~), default derived from ~scope:~ opt;
     ~__hostable__/0~ introspection.
   - transformer ~Concept.Hostable.Transformers.InstallConversations~ (like
     ~AutoTools.SynthesizeTools~): register host in a config-backed registry
     (feeds ~Conversation.host_type~ one_of); add ~has_many :conversations~
     filtered to this host_type; synthesize the described ~discuss~ update
     action (so AutoTools projects it to MCP — outcome is FREE).
   - ~Concept.Hostable.registered/0~ reads the registry for the ~one_of~.
   - ~resolve_scope/2~: :subtree→"page:<id>", {:self,_}→"record:<id>",
     :workspace→:workspace, {:union,refs}→passthrough.
2. ~use Concept.Hostable, scope: :subtree~ on ~Concept.Pages.Page~ only.
3. ~Conversation~: add ~host_type :atom~ (one_of: Hostable.registered),
   ~host_id :uuid~ (nil ⇔ :workspace). Migration backfills existing rows to
   ~host_type: :workspace, host_id: nil~. DROP the 1:1 reflex:
   ~belongs_to :user, allow_nil?: false~ + ~relate_actor(:user)~ +
   ~my_conversations~ filter ~user_id == actor.id~ (replace with host-scoped read).
4. ~discuss~ action resolver (3-arm, but only find-or-create + post arms this
   wave): resolve the host's default conversation, post a message. Plain change
   now; Reactor deferred to when thread-spawn lands (Wave 3).
5. Point the existing ChatPanel at a Page-hosted conversation; confirm RAG
   scopes to the page via the EXISTING ~source_id: "page:<id>"~ path (no new
   retrieval code).

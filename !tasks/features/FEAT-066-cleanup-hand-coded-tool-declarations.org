* Intention & Purpose

Remove ~tool ...~ declarations from domain files that are now redundant
because ~Concept.AutoTools~ (FEAT-052/053) synthesizes them from the
described actions. Single source of truth.

Delegable to ~quick_task~ — purely mechanical removal with verification.

* Context (read in full)

** Why this is safe

After W0–W3:
- Every described action is exposed via AutoTools.
- The parity test (FEAT-063) asserts every described action appears in
  ~tools/list~.

A manual ~tool :name, Resource, :action do description "..."~ entry
duplicates what AutoTools would synthesize. AutoTools' transformer
detects the manual entry by name and *prefers* it (with a compile
warning). Removing the manual entries silences the warning and centers
the contract on the action itself.

** Where the entries live

- ~lib/concept/pages.ex~ — one entry: ~:create_page~.
- ~lib/concept/knowledge.ex~ — three entries: ~:search_workspace~,
  ~:answer_question~, ~:link_blocks~.
- ~lib/concept/knowledge/chat.ex~ — two entries:
  ~:chat_list_conversations~, ~:chat_message_history~.

* Scope (IN)

** Files (Edit)

- ~lib/concept/pages.ex~ — drop the entire ~tools do .. end~ block.
- ~lib/concept/knowledge.ex~ — drop the entire ~tools do .. end~ block.
- ~lib/concept/knowledge/chat.ex~ — drop the entire ~tools do .. end~ block.

* Before each deletion (verification step)

For each manual ~tool :foo, Resource, :action~ being removed:

1. Confirm ~Resource.action(:action)~ has a non-nil ~description~ (else
   removing the manual entry will unexpose it).
2. Confirm the synthesized name matches what consumers expect. The
   original name vs auto-synthesized name:

   | Original (manual)         | Auto-synthesized          | Match? |
   |---------------------------+---------------------------+--------|
   | ~:create_page~            | ~:page_create_page~       | NO     |
   | ~:search_workspace~       | ~:tools_search_workspace~ | NO     |
   | ~:answer_question~        | ~:tools_answer_question~  | NO     |
   | ~:link_blocks~            | ~:link_create~            | NO     |
   | ~:chat_list_conversations~| ~:conversation_my_conversations~ | NO |
   | ~:chat_message_history~   | ~:message_for_conversation~ | NO  |

   Names DO change. Two options:

   (a) Accept the name change. Communicate via release notes; update any
       client that hardcoded these names.

   (b) Keep manual entries as *aliases* — change them to use the new
       AutoTools-synthesized name as the canonical and keep the old name
       only if a documented external client depends on it.

   *Default decision*: (a). The MCP client list is small (internal Spell
   harness + maybe a few local MCP inspectors); a rename is cheap and
   the new naming is more consistent across the surface.

* Implementation steps

1. Delete the three ~tools do .. end~ blocks.
2. ~mix compile --warnings-as-errors~ — expect no warnings about
   "manual tool shadows auto-synthesized".
3. ~mix test --exclude integration~.
4. ~mix concept.docs.mcp_surface~ — regenerate the docs; verify tool
   names update.
5. Run the parity test: ~mix test test/integration/mcp_parity_test.exs~.

* Initial Message

how would you integrate Arcana into this, maximally leveraging Ash to make the system as simple as possible?

* Intention & Purpose
Introduce a single Ash domain `Concept.Knowledge` that becomes the ONLY entry point into Arcana from the rest of the app. Every later item (ingestion, search, chat, AI block, dashboard) calls into this domain so policies, telemetry, and tenancy live in one place and Arcana's own schemas remain encapsulated.

* Context & Narrative
Concept exposes two existing domains: `Concept.Accounts` (User/Workspace/Membership) and `Concept.Pages` (Page/Block). Workspace tenancy is enforced via `multitenancy strategy: :attribute, attribute: :workspace_id`. Arcana itself does NOT respect Ash multitenancy — tenant isolation is enforced solely by collection name. Wrapping it in an Ash domain with policies = correctness floor.

This item creates the empty-but-correct domain shell. Generic actions are declared with placeholder implementations (returning `{:error, :not_implemented}`); subsequent items fill them in. This lets us land the routing/policy/structure first and verify it compiles + passes tests before the heavy lifting.

* Scope
- IN: `Concept.Knowledge` Ash domain module; placeholder generic actions `:ingest_page`, `:delete_page`, `:search`, `:ask`, `:rebuild_workspace_graph`, `:rebuild_communities`, `:evaluate_ai_block`; policy that all callers must be a workspace member; domain registered in `config :concept, ash_domains` list.
- OUT: actual Arcana calls (filled in by FEAT-022/023/024/028); UI; telemetry attach.

* Existing Patterns (MUST follow)
- `lib/concept/pages.ex` & `lib/concept/accounts.ex` define `use Ash.Domain, otp_app: :concept, extensions: [AshAdmin.Domain]` with `admin do show? true end`. Mirror exactly.
- Workspace-membership check: `lib/concept/pages/checks/workspace_member.ex` (referenced by Pages domain). Reuse it — do not re-implement.
- Generic actions go through `Ash.ActionInput` and `Ash.Domain.Info.run_action/2`. Use Ash 3.x style: `actions do generic :name do ... end end`.
- `config/config.exs` will already contain `config :concept, ash_domains: [...]`. Locate this list (likely set elsewhere; grep first) and add `Concept.Knowledge` as a sibling to `Concept.Accounts` and `Concept.Pages`.
- `Concept.Pages.Checks.WorkspaceMember` operates on the actor and the loaded record's workspace_id. For domain-level generic actions there is no record — use `Ash.Policy.Check.Builtins.actor_present()` plus a custom check that loads `Membership` for the supplied `workspace_id` argument.

* Tests (file paths + scenarios)
Test files: `test/concept/knowledge_test.exs`, `test/concept/knowledge/policy_test.exs`.
Scenarios:
1. `Concept.Knowledge` is listed in `Application.fetch_env!(:concept, :ash_domains)`.
2. Calling `:ask` action without an actor → `Ash.Error.Forbidden`.
3. Calling `:ask` action with an actor who is NOT a member of the workspace → `Ash.Error.Forbidden`.
4. Calling `:ask` action with a workspace member actor → returns `{:error, :not_implemented}` (or whatever placeholder reaches us; later overridden).
5. Domain reflection: `Ash.Domain.Info.resources(Concept.Knowledge)` returns `[]` (we hold no resources, only generic actions).
6. Each declared action is enumerable via `Ash.Domain.Info.actions(Concept.Knowledge)` and named correctly.

  ```elixir
  defmodule Concept.Knowledge do
    @moduledoc """
    RAG/GraphRAG over Concept's pages & blocks. Wraps Arcana with
    workspace tenancy + Ash policies.
    """
    use Ash.Domain, otp_app: :concept, extensions: [AshAdmin.Domain]

    admin do
      show? true
    end

    actions do
      generic :ingest_page, returns: :map do
        argument :page_id, :uuid, allow_nil?: false
        argument :workspace_id, :uuid, allow_nil?: false
        run fn _input, _ctx -> {:error, :not_implemented} end
      end

      generic :delete_page, returns: :atom do
        argument :page_id, :uuid, allow_nil?: false
        argument :workspace_id, :uuid, allow_nil?: false
        run fn _, _ -> {:error, :not_implemented} end
      end

      generic :search, returns: {:array, :map} do
        argument :query, :string, allow_nil?: false
        argument :workspace_id, :uuid, allow_nil?: false
        argument :limit, :integer, default: 10
        run fn _, _ -> {:error, :not_implemented} end
      end

      generic :ask, returns: :map do
        argument :question, :string, allow_nil?: false
        argument :workspace_id, :uuid, allow_nil?: false
        argument :reply_to, :term, allow_nil?: true
        argument :scope, :map, default: %{kind: :workspace}
        run fn _, _ -> {:error, :not_implemented} end
      end

      generic :rebuild_workspace_graph, returns: :map do
        argument :workspace_id, :uuid, allow_nil?: false
        run fn _, _ -> {:error, :not_implemented} end
      end

      generic :rebuild_communities, returns: :map do
        argument :workspace_id, :uuid, allow_nil?: false
        run fn _, _ -> {:error, :not_implemented} end
      end

      generic :evaluate_ai_block, returns: :map do
        argument :block_id, :uuid, allow_nil?: false
        argument :workspace_id, :uuid, allow_nil?: false
        run fn _, _ -> {:error, :not_implemented} end
      end
    end

    policies do
      bypass actor_attribute_equals(:system?, true) do
        authorize_if always()
      end

      policy always() do
        authorize_if Concept.Knowledge.Checks.WorkspaceArgumentMember
      end
    end
  end
  ```

* Edge Cases
- A user passes a `workspace_id` for a workspace they do not own AND of which they are not a member → must fail with `Ash.Error.Forbidden` (NOT `:not_implemented`). Policy must run before action.
- Anonymous actor → forbidden.
- `system?` actor (used by background workers) → bypass policy (mirrors patterns in Pages domain).
- Renaming actions later breaks policy specs and AshAdmin admin pages — document the names as part of the public API of the domain.

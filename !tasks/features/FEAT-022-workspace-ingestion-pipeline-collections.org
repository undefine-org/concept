* Initial Message

how would you integrate Arcana into this, maximally leveraging Ash to make the system as simple as possible?

* Intention & Purpose
Make Concept's content searchable. On every page/block create/update/destroy, schedule a debounced background job that re-ingests the affected page into Arcana under the per-workspace collection. This fully implements `Concept.Knowledge.ingest_page` and `:delete_page`.

* Context & Narrative
Concept already runs Oban + AshOban. `Concept.Pages.Page` and `Concept.Pages.Block` resources need a `notifiers` clause that triggers an AshOban-scheduled worker. Arcana's ingest is synchronous and can take seconds for a large page; therefore it MUST run in a background job.

We adopt this strategy:
1. **Per-page job key**: `{:ingest_page, page_id}` is the unique-args key. Pending jobs collapse via Oban's `unique` constraint with `replace: [scheduled: [:scheduled_at]]` — successive saves within the debounce window do not duplicate jobs; the latest save's `scheduled_at` wins.
2. **Debounce**: schedule for `now + 2s`. Multiple updates within 2s collapse.
3. **Block change → page resolution**: a block's `page_id` is used as the job key. Reparenting a block from page A to page B emits TWO jobs (one for each page) via two notifier invocations.
4. **Delete cascades**: archiving a page enqueues a delete job; on completion no chunks remain.
5. **Tenant context**: workers run with `actor: %SystemActor{system?: true}, tenant: workspace_id` to bypass member policies.

- `ingest_page(page_id, workspace_id)`:
  1. Load Page via `Concept.Pages.get_page!(page_id, tenant: workspace_id, actor: %SystemActor{})`.
  2. Load all non-archived blocks via `Concept.Pages.list_for_page(page_id, tenant: workspace_id, actor: %SystemActor{})`.
  3. Compute breadcrumbs by walking parent_page_id.
  4. Ensure collection.
  5. `delete_page(page_id, workspace_id)` to remove prior chunks.
  6. Render full page markdown (concat of block markdowns) for the document body.
  7. Call `Arcana.ingest(markdown, repo: Concept.Repo, collection: name, source_id: "page:#{page_id}", metadata: %{...}, chunker: Concept.Knowledge.BlockChunker, chunker_opts: [blocks: blocks, page: page, workspace_id: ws_id, breadcrumbs: bc], embed_opts: [intent: :document])`.
- `delete_page(page_id, workspace_id)`:
  1. Look up collection by name.
  2. `Concept.Repo.delete_all(from d in Arcana.Document, where: d.collection_id == ^col.id and d.source_id == ^"page:#{page_id}")` — chunks cascade via FK on_delete.

* Edge Cases
- Page archive while job pending: worker loads via Pages action which returns `nil` due to `archive base_filter? true`; worker treats as deletion (calls `delete_page`), returns `{:ok, :archived}`.
- Block in archived state: `Block.archive` filters from queries; chunker only sees live blocks (correct).
- Ingestion failure mid-batch (Gemini transient): job retries; previous chunks already deleted ⇒ search returns empty during retry window. Acceptable.
- Notifier fires inside a transaction that later rolls back: AshOban `notify_via :pubsub` mode is in-transaction-safe (uses Oban's transactional insert — Oban must be configured with same Repo, which it is).
- Multi-node deployment: Oban distributes per-node; `unique` constraint at Postgres prevents duplicate jobs across nodes.
- Lock state changes (Block locks) trigger needless reindex: skip in notifier when only `lock_state | lock_holder_id | lock_acquired_at | lock_expires_at` changed (use `Ash.Resource.Change.in_diff?`).

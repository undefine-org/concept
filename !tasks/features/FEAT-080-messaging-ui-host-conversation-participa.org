* Goal
The human-facing surface for the host-conversation model. Mockups in
docs/messaging_design.md (host conversation; crystallize). LiveView code-interface
ONLY (Credo LiveViewPurity EX9001 — no Ash.Query in lib/concept_web/).

* Scope
1. Host-conversation view: the document blocks and the thread are ONE continuous
   surface ("Conversation about this page" divider). Append-only message stream
   with mixed senders.
2. Participant rail: humans (avatars), external agents (violet robot badge),
   and the host voice ("This page" / "This task", violet sparkle — NOT a person).
3. @-mention composer: registry dispatch (member → notify chip; hostable →
   link chip); ~/~ slash menu so a message can contain blocks (reuse editor).
4. Crystallize UI: select messages → "Crystallize N → page"; promoted blocks
   show green "just added" + provenance chip "from conversation".
5. Inbox LiveView: subscribes to ~inbox:<user_id>~; lists unread conversations,
   each context-rich (shows its host). Agent results + human mentions in one list.
6. Threads: a message can spawn/show a child thread (nested).

* Constraints
- All data access via domain code-interface fns (define in the Chat/Pages
  domains); NO ~Ash.Query~/~Ash.Changeset.for_*~/~Ecto.Query~ in web layer.
- Real-time via existing PubSub topics + the new ~inbox:~ topic.
- Reuse ChatPanel / BlockRender / slash menu; do not fork the editor.

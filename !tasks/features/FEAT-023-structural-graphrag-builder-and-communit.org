* Initial Message

how would you integrate Arcana into this, maximally leveraging Ash to make the system as simple as possible?

* Intention & Purpose
Concept's data is a graph: pages contain pages and blocks; blocks contain blocks; users author them. We feed THAT structure directly into Arcana's GraphRAG schemas — bypassing LLM-based entity extraction entirely. Result: free, deterministic, complete graph that powers Arcana's Fusion Search (vector ∪ graph traversal) for richer retrieval.

This is the riskiest item: we write to Arcana's *internal* schemas (`Arcana.Graph.Entity`, `Arcana.Graph.Relationship`) without a public API. We pin the version (`~> 2.0`) and add a regression test that asserts on schema fields so a major Arcana upgrade fails loudly.

* Context & Narrative
Arcana's default GraphRAG flow extracts entities + relations via an LLM — costly, lossy, redundant when the structure is already explicit in the source data. Microsoft's GraphRAG paper shows that *community summaries* (LLM-summarize each cluster) materially improve global-question quality. We retain this single LLM-driven step but make it on-demand: weekly via AshOban OR manual button in workspace settings.

The structural graph captures:

```
Workspace --HAS_PAGE-->     Page
Page      --PARENT_OF-->    Page         (parent_page_id)
Page      --CONTAINS  -->   Block        (page_id, parent_block_id IS NULL)
Block     --PARENT_OF-->    Block        (parent_block_id)
Block     --HAS_TYPE -->    BlockType    (atom-keyed entity)
Block     --AUTHORED_BY-->  User         (paper_trail.created_by? — defer if missing)
```

Each `Arcana.Graph.Entity` row gets `kind: "page" | "block" | "block_type"`, `name: title or block.type`, `metadata: %{breadcrumbs, page_id, ...}`, `collection_id` of the workspace.

* Scope
- IN: per-page graph upsert called from `ingest_page` AFTER chunking; workspace-level `rebuild_workspace_graph` action (full rebuild from Pages); `rebuild_communities` action invoking Arcana's community-detection + LLM summary; AshOban scheduled job (weekly) + manual trigger; workspace settings button (owner-only); regression tests against schema shape.
- OUT: Block→Page LINKS_TO edges based on Lexical mentions (defer to FUP — needs Lexical mention spec in Concept).

Scenarios:

* Edge Cases
- Arcana 2.x bumps schema fields: regression test catches it; GraphBuilder is the SINGLE place that needs patching.
- Block authored_by: paper_trail captures `created_by` only if Concept stores it — currently `paper_trail` is in deps but no `actor:` capture is wired on Block creation. AUTHORED_BY edges are skipped in v1; FUP to wire actor capture.
- Empty workspace (no pages): `rebuild_workspace_graph` is a no-op returning `{:ok, %{entities: 0, relationships: 0}}`.
- Concurrent ingest of same page: Postgres unique constraint on conflict_target serializes upserts; later wins. Acceptable.
- Community summary cost: each summary is one LLM call; a workspace with 50 communities = 50 calls. Worker uses `:knowledge_communities` queue with concurrency=1 to throttle.
- Community summary mid-run + page edit: the next `rebuild_communities` will reflect the new edges. No locking needed.
- Arcana lacking `summarize_communities` (or different name): fall back to Markdown-style "list of nodes in community" stored in summary field — keeps the system functional even if the public API is incomplete.

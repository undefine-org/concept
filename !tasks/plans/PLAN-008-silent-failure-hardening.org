* PLAN Silent Failure Hardening

** Premise

Every silent bug we hit this week shared one shape:

#+begin_quote
A boundary between two systems is wired by string/atom name (not by a
typed reference), and one side fails open (no error, no log) when the
names don't match.
#+end_quote

We can't catch these with linters or types \u2014 the boundaries cross
processes (Lit \u2194 LV, PubSub topic \u2194 subscriber) or are inherently
runtime-dispatched (Ash code interface \u2194 resource action). The fix is
structural: *one round-trip test per boundary class*, generic where
possible, with the property that any silent miswire fails the suite.

This plan installs that backstop.

** Boundary inventory

|--------------------------------+------------------------------------------------+----------------------+-------------|
| Boundary                       | Failure mode                                   | Today                | Caught by   |
|--------------------------------+------------------------------------------------+----------------------+-------------|
| Lit dispatch \u2192 LC handle_event | typo'd CustomEvent name silently no-ops        | per-block tests      | T-1         |
| PubSub topic \u2192 subscriber      | wrong bus or wrong topic shape \u2192 no delivery   | one fixed in EvalAI  | T-2         |
| Ash action \u2192 permission probe  | validation crashes on `Ash.can?(_, %{})`       | one fixed in Link    | T-3         |
| Ash code interface \u2192 resource  | argument silently ignored (set_argument order) | none (one prod fix)  | T-4         |
| Ash actor \u2192 resource policy    | bare map fails relate_actor                    | none                 | T-5         |
| Ash changeset state contract   | warnings ignored in test logs                  | logger warnings live | T-4 (lever) |

T-1 is already in place (the AI Answer contract test from PLAN-006). T-2
through T-5 are new.

** What this is not

- Not a refactor. No production code changes \u2014 only tests + one Logger
  config + one Credo check.
- Not a "catch everything" tool. Each test pins one boundary class. A new
  boundary class would need its own pinning.
- Not a replacement for integration tests. These are sharp regressions
  for known failure shapes; they coexist with end-to-end tests.

** Phases

*** P-1 \u2014 Ash can?-probe across every action (T-3)

The Link bug was: AshAI calls `Ash.can?({Concept.Knowledge.Link, :create, %{}}, user)` to build its tool registry, and the cross-block validation crashed on `Concept.Repo.get(_, nil)`. Same shape can hit any action with a cross-attribute validation that doesn't nil-guard.

**** T-3.1 \u2014 ⊕ ~test/concept/ash_can_probe_test.exs~

#+begin_src elixir
defmodule Concept.AshCanProbeTest do
  @moduledoc """
  Every action across every domain must survive `Ash.can?/3` with an empty
  input. AshAI's tool registry builder calls this on each action; any
  validation that crashes on missing attrs (e.g. `Ecto.Repo.get(_, nil)`)
  takes the entire respond pipeline down.

  Add `only_when_valid?: true` to the offending change, or nil-guard the
  validation body.
  """
  use Concept.DataCase, async: false

  @domains [
    Concept.Pages, Concept.Knowledge, Concept.Knowledge.Chat, Concept.Accounts
  ]

  setup do
    {:ok, user} =
      Concept.Accounts.User
      |> Ash.Changeset.for_create(:register_with_password, %{...})
      |> Ash.create(authorize?: false)
    %{user: user}
  end

  for domain <- @domains do
    for resource <- Ash.Domain.Info.resources(domain) do
      for %{name: action_name, type: type} when type in [:create, :update, :destroy]
            <- Ash.Resource.Info.actions(resource) do
        test "Ash.can?({#{inspect(resource)}, :#{action_name}, %{}}) does not raise",
             %{user: user} do
          assert Ash.can?(
                   {unquote(resource), unquote(action_name), %{}},
                   user,
                   run_queries?: false
                 ) in [true, false]
        end
      end
    end
  end
end
#+end_src

Discovers actions via ~Ash.Resource.Info.actions/1~; no maintenance as new actions land. Likely produces ~40\u201380 tests, runs in <2s, catches the Link-class bug for the entire codebase.

**** T-2.1 \u2014 ⊕ ~test/concept/pubsub_contract_test.exs~

The contract: any resource declaring `pub_sub do module M` must have all its broadcasts subscribed via M, not a different bus. Today we have one resource per module choice (~ConceptWeb.Endpoint~ for Chat, ~Concept.PubSub~ for blocks). Cross-checking is feasible:

#+begin_src elixir
test "every Ash pub_sub broadcaster has at least one matching subscriber" do
  broadcasters = scan_resources_for_pub_sub()        # %{Resource => Endpoint module}
  subscribes  = scan_for_subscribes()                # [{file, target_module, topic_pattern}]

  for {resource, expected_mod} <- broadcasters do
    matching = Enum.filter(subscribes, fn {_, mod, topic_pat} ->
      mod == expected_mod and topic_pat =~ topic_root_for(resource)
    end)

    assert matching != [], """
    Resource #{inspect(resource)} broadcasts via #{inspect(expected_mod)}
    but no subscriber listens. Either remove the pub_sub block or add a
    matching subscriber.
    """
  end
end
#+end_src

This is a code-scan test (AST walk over ~lib/**~). About 60 LOC. Catches the EvaluateAi-class bug: subscribing to ~Concept.PubSub~ while the resource broadcasts via ~ConceptWeb.Endpoint~.

*** P-3 \u2014 Ash.Changeset misuse fatal in test (T-4)

The set_argument-after-for_create silent drop is the worst class: Ash 3.x logs `[warning] Changeset has already been validated for action :create` and *silently ignores* the call. This is fixable two ways:

**** T-4.1 \u2014 ⊕ test/support/ash_warning_filter.ex

Logger backend that fails the current test if any `[warning] Changeset has already been validated` or `[warning] force_change_attribute after for_create` shows up during the test. Configured only in ~config/test.exs~.

#+begin_src elixir
defmodule Concept.Test.AshWarningFilter do
  def attach do
    :logger.add_handler(:ash_warning_fatal, __MODULE__, %{level: :warning, formatter: ...})
  end

  def log(%{level: :warning, msg: msg}, _state) do
    text = IO.iodata_to_binary(msg)

    fatal_patterns = [
      "Changeset has already been validated for action",
      "force_change_attribute after for_create"
    ]

    if Enum.any?(fatal_patterns, &String.contains?(text, &1)) do
      raise "Forbidden Ash warning in test: #{text}"
    end

    :ok
  end
end
#+end_src

Wire in ~test_helper.exs~. Now any test that exercises the buggy ordering fails loudly.

**** T-4.2 \u2014 Optional follow-on: Credo check banning the syntactic pattern

~|> Ash.Changeset.set_argument~ appearing AFTER ~|> Ash.Changeset.for_*~ in the same pipeline is always wrong. A Credo plugin (or simpler: a regex-based ~mix~ task) catches the bug at compile time. ~60 LOC. Lower priority; T-4.1 already covers the runtime symptom.

**** T-5.1 \u2014 ✎ ~lib/concept/pages/block_type/interactive.ex~

def update(assigns, socket) do
  unless match?(%Concept.Accounts.User{}, assigns[:current_user]) or
         match?(%{system?: true}, assigns[:current_user]) do
    raise ArgumentError, """
    #{inspect(__MODULE__)} requires `current_user` to be a
    `Concept.Accounts.User` struct or `%{system?: true}`, got:

        #{inspect(assigns[:current_user])}

    This is the LiveComponent actor contract for Interactive blocks. If
    you see this in a nested live_render, load the User struct in mount/3.
    """
  end
  ...
end
#+end_src

User-overridable `update/2` can call `super(assigns, socket)` to invoke the guard. Future Interactive blocks crash at mount, not deep in Ash. Documented contract.

** Acceptance

1. ~mix precommit~ stays green.
2. Reverting any of the past week's fixes (Link nil-guard, EvaluateAi subscribe, set_argument order, RequireOwnLock system actor) makes the suite fail with a clear error pointing at the specific class.
3. New silent bugs of these classes can be reproduced by writing a one-liner that hits the boundary; the round-trip test will fail.
4. ~docs/blocks/ADDING_A_BLOCK.md~ documents the actor contract.

** Non-goals

- Generic "catch all silent failures" tooling. Each boundary still needs explicit handling.
- Migrating away from string-named protocols. Atom IDs and topic names stay; we add the round-trip backstop instead.
- Catching JS-side typos (FUP for `OraBlock`'s `data-events` runtime warn covers that separately).

How would you make it so that any feature we develop on the frontend is readily accessible (for free) to LLMs via the MCP? Maximally use Ash. Ideally, frontends are just the 'human connection' layer and Ash and its ecosystem lets us have the MCP 'for free' that lets you use Concept as easily as a human would.

* PLAN MCP Parity by Construction
** Principle

#+begin_quote
Every Ash action with a ~description~ is a feature. The LiveView is one
projection of that feature; the MCP tool is another. If a human can do it
in Concept, an LLM can do it via MCP — automatically, with the same
policies, on the same data.
#+end_quote

This plan makes the principle mechanically enforced. After it lands:

- Adding a feature ::= adding an Ash action with a ~description~.
- LiveView wiring is mechanical; MCP exposure is automatic; tests prove parity.
- Block types declare their MCP surface in the same place they declare LV + JS wiring (via ~BlockType.{Static,Interactive}~).
- Documentation generates from action metadata (~mix concept.docs.mcp_surface~).

** Why now

Today only 4 of ~30 actions are MCP-exposed (whitelist in router). The
infrastructure (AshAi 0.6.1, AshAuthentication ApiKey strategy, policy
authorizer, multitenancy) is already in place. The bottleneck is
*declaration* and an *enforcement loop*. PLAN-006 just established the
"one module = one feature" pattern for block types — this plan generalizes
it to the whole product.

** Architecture

#+begin_example
Ash action (the canonical unit)
    │
    ├── description "..."          ⇒ AutoTools → MCP tool
    ├── arguments + types          ⇒ MCP JSON schema (free)
    ├── policies                   ⇒ enforced on every surface
    ├── pub_sub                    ⇒ LiveView re-render
    │
    └── projections:
          • LiveView handle_event/3 → calls code interface
          • AshPhoenix.Form         → same action
          • MCP /mcp tools/call     → same action
          • MCP /mcp resources/read → curated read actions
          • Block-type macros       → register tool + ui_resource per verb
#+end_example

The MCP server stops being a separate surface and becomes a *projection of
domain actions*. Same authz, same tenancy, same data path.

** Risks & mitigations

- *Auto-exposing mutation actions* — every action must have a ~policies~
  block. Audit policies in W1 (descriptions writer also asserts a policy
  block exists). FEAT-058 ships before FEAT-060 so workspace context is
  required before the whitelist drops.
- *Description drift* — mitigated by the mix-task assertion; CI red on miss.
- *Argument schema quality* — typed args (UUID, atom one_of, string) yield
  good JSON schemas. Free-form ~:map~ args (e.g. ~Block.update_props~)
  need their shape described in prose. W1 enforces this per arg.
- *Block-type macro feature creep* — keep FEAT-064/065 additive; no breaking
  changes to the ~ash_actions~ syntax in ~ADDING_A_BLOCK.md~. New behavior
  is gated by a new optional ~mcp:~ keyword on ~use~.

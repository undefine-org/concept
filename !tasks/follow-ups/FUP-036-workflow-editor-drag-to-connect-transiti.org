Deferred from the Objects editor thread (E2). The workflow editor ships a **list-first** transition editor (from→to selects + guard palette). The original UX vision (docs/objects_and_tasks_ux.md §3) imagined a visual graph canvas where states are nodes and transitions are drawn by dragging between them.

Scope: a node/edge canvas (states as draggable nodes, drag-from-handle-to-node to create a transition), replacing or augmenting the list view. Likely a JS hook (e.g. a lightweight graph lib or hand-rolled SVG) pushing the same `add_transition` event the list form pushes — zero new server authority, same `create_transition` code-interface.

Non-goal: changing the domain. The list editor already exercises 100% of the capability; this is pure UX polish.

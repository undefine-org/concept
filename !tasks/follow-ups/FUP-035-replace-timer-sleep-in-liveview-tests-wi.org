* Observation (2026-05-29, false-green sweep)

Several LiveView/component tests use =:timer.sleep/1= to wait for async work,
violating the project test guideline ("Avoid Process.sleep; use
Process.monitor / :sys.get_state"). Sleeps are flaky (pass/fail by timing) and
slow the suite. Remaining offenders after the BUG-065 sweep:

- test/concept_web/live/ask_selection_test.exs
- test/concept_web/live/lock_indicator_test.exs
- test/concept_web/live/workspace_chat_panel_test.exs (palette_ask test)
- test/concept_web/components/live_citation_rail_test.exs
- test/concept_web/components/indexing_pill_test.exs

(Already cleaned: the scope/profile dropdown tests in workspace_chat_panel.)

* Decision

Replace each =:timer.sleep= with a deterministic barrier:
- after a =send(view.pid, msg)=: =_ = :sys.get_state(view.pid)= to flush the
  mailbox before asserting.
- for assign_async work: =render_async(view)=.
- for PubSub round-trips: assert_receive on the broadcast, then render.

* Scope
- The 5 test files above. No production code.

* Acceptance
- No =:timer.sleep= / =Process.sleep= remain under test/concept_web/.
- Tests deterministic; suite green across repeated --seed runs.

# MCP Parity by Construction

> Every Ash action with a `description` is a feature.
> The LiveView is one projection of that feature; the MCP tool is another.
> If a human can do it in Concept, an LLM can do it via MCP —
> automatically, with the same policies, on the same data.

This doc explains the principle that governs how Concept exposes
functionality to humans and to LLMs. Read it once; the rules in
[`AGENTS.md`](../AGENTS.md) are the day-to-day reference.

For the live, auto-generated list of every MCP tool and resource, see
[`docs/mcp_surface.md`](mcp_surface.md) (regenerated by
`mix concept.docs.mcp_surface`).

## Why this matters

Most products bolt MCP support on after the fact: an extra surface, an
extra mapping layer, an extra set of bugs. Two kinds of code that
diverge silently the moment anyone adds a feature in only one place.

Concept does not have that problem. The Ash action is the canonical
unit. LiveView is one projection. MCP is another. AshAdmin is a third.
They cannot disagree because they read the same source.

Add a feature once. The LLM gets it on the same commit.

## Architecture

```
Ash action (the canonical unit)
    │
    ├── description "..."          ⇒ AutoTools → MCP tool
    ├── arguments + types          ⇒ MCP JSON schema (free)
    ├── policies                   ⇒ enforced on every surface
    ├── pub_sub                    ⇒ LiveView re-render
    │
    └── projections:
          • LiveView handle_event/3 → calls code interface
          • AshPhoenix.Form         → same action
          • MCP /mcp tools/call     → same action
          • MCP /mcp resources/read → curated read actions
          • Block-type macros       → register tool per verb
```

The MCP server is not a separate surface; it is a projection of domain
actions. Same authz, same tenancy, same data path.

## The action is the unit

An Ash action becomes an MCP tool when three conditions hold:

1. It is a **public** action (declared inside `actions do ... end`).
2. It has a non-nil **`description`**.
3. It is not in the **opt-out list** (see below).

The mechanism is `Concept.AutoTools`, a Spark extension wired into every
domain alongside `AshAi`. Its single transformer walks each domain's
resources at compile time and synthesizes one `%AshAi.Tool{}` entity per
described action. The synthesized name is `:"<resource_slug>_<action_name>"`,
e.g. `Concept.Pages.Page :rename` → `:page_rename`.

If a domain author writes a manual `tool ...` entry whose name collides
with what would be synthesized, the manual entry wins and a compile
warning is emitted. The principle is single-source-of-truth: prefer the
action's description.

## Description style

The description doubles as the LLM-visible tool documentation **and**
the human-facing docstring in ExDoc/AshAdmin. Treat it as both.

- **One sentence**, < 110 chars where possible.
- **Imperative voice for mutations** ("Rename a page.").
- **Declarative for reads** ("List a workspace's pages…").
- **Domain language**, no implementation jargon.
- Per-argument descriptions describe meaning + constraints.

Good:

```elixir
update :rename do
  description "Rename a page. Title is trimmed of leading and trailing whitespace."
  accept [:title]
end

argument :workspace_id, :uuid,
  allow_nil?: false,
  description: "Workspace to create the page in. Must be a workspace the actor is a member of."
```

Bad:

```elixir
update :rename do
  description "Calls trim_title change and updates the title attribute via the changeset."
  accept [:title]
end
```

The second leaks implementation detail. The user (human or LLM) does not
know what `trim_title` is or care about changesets.

## Per-argument descriptions

LLMs construct tool calls from JSON schemas, which AshAi derives from
each argument's type + description. A missing or vague description leads
to wrong calls.

Hard rule: every `public?: true` argument on a described action carries
a `description:`. The integration parity test asserts this in CI.

## MCP resources vs tools

Two MCP surfaces:

- **Tools** (`tools/list` + `tools/call`) — verbs. The LLM invokes them.
  Map 1:1 to Ash actions via AutoTools.
- **Resources** (`resources/list` + `resources/read`) — nouns. The LLM
  *browses* them like a human reads the sidebar.

Use a tool for *do this thing*. Use a resource for *show me what's
here*. Resources are addressable URIs (`concept://me/workspaces`)
backed by generic Ash actions returning string-serialized payloads
(JSON, HTML, plain text). The verifier
`AshAi.Verifiers.McpResourceActionsReturnString` enforces the return
type.

## Workspace tenancy on MCP

`ConceptWeb.Plugs.MCPWorkspaceContext` resolves the workspace tenant
once per request, before any action runs. The resolution ladder:

1. **ApiKey is workspace-bound** (the API key carries `workspace_id`) →
   that workspace. **The `mcp-workspace-id` header is ignored.** Bound
   keys are stronger than per-request hints — this prevents privilege
   confusion.
2. **Header `mcp-workspace-id`** + the actor is a member of that
   workspace → that workspace.
3. **Neither** → no tenant assigned. Workspace-agnostic actions still
   run (e.g. `workspace_for_user`); mutation tools on tenanted resources
   refuse via the resource's tenancy invariant.

| ApiKey.workspace_id | mcp-workspace-id header | Member of header's WS? | Resolved tenant |
|---|---|---|---|
| `WS_A` | (any value, ignored) | n/a | `WS_A` |
| `nil` | `WS_B` | yes | `WS_B` |
| `nil` | `WS_C` | no | HTTP 400 (`not_a_member`) |
| `nil` | absent | n/a | `nil` (agnostic only) |

The plug stores the resolved value in both `conn.assigns[:mcp_tenant]`
and `Ash.PlugHelpers.set_tenant/2` so downstream actions inherit it
without manual threading.

## Block types

PLAN-006 established that each block type is a single module that owns
its full vertical slice: schema, validation, render, slash-menu entry,
event handlers. PLAN-007's FEAT-064 added a fifth slice to the same
module: MCP exposure.

```elixir
use Concept.Pages.BlockType.Interactive,
  ash_actions: [
    evaluate: [Concept.Pages, :evaluate_ai, [:prompt, :scope, :profile]],
    refresh:  [Concept.Pages, :evaluate_ai, [:prompt, :scope, :profile]],
    retry:    [Concept.Pages, :evaluate_ai, [:prompt, :scope, :profile]]
  ],
  mcp: [
    descriptions: [
      evaluate: "Run the AI Answer block's prompt and stream cited results.",
      refresh:  "Re-run the prompt against the current workspace state.",
      retry:    "Re-run the AI Answer after a failure."
    ]
  ]
```

Each `ash_actions:` verb is synthesized as a distinct MCP tool named
`block_<type>_<verb>` (e.g. `block_ai_answer_evaluate`). All three verbs
above point to the same underlying Ash action (`Block :evaluate_ai`);
the verb-specific descriptions help the LLM pick the right one for its
intent.

Opt-out is per-verb via `mcp: [except: [:retry]]` or
`mcp: [only: [:evaluate]]`. Omitting `mcp:` exposes every verb with
auto-generated descriptions.

UI resources (rendering the block visually for MCP clients that support
MCP Apps) require AshAi support beyond what 0.6.1 offers; tracked in
FUP-033.

## Anti-patterns

### Red: LiveView with inline Ash query

```elixir
def handle_event("show_recent", _, socket) do
  jobs =
    Concept.Knowledge.IngestionJob
    |> Ash.Query.sort(inserted_at: :desc)
    |> Ash.Query.limit(10)
    |> Ash.read!(actor: ..., tenant: ...)

  {:noreply, assign(socket, jobs: jobs)}
end
```

### Green: domain code-interface fn

In `lib/concept/knowledge/ingestion_job.ex`:

```elixir
read :recent_for_workspace do
  description "List the 10 most recent ingestion jobs in the workspace."
  prepare build(sort: [inserted_at: :desc], limit: 10)
end
```

In `lib/concept/knowledge.ex`:

```elixir
resource Concept.Knowledge.IngestionJob do
  define :recent_ingestion_jobs, action: :recent_for_workspace
end
```

In the LiveView:

```elixir
def handle_event("show_recent", _, socket) do
  jobs = Concept.Knowledge.recent_ingestion_jobs!(actor: ..., tenant: ...)
  {:noreply, assign(socket, jobs: jobs)}
end
```

The action is reachable from the LiveView **and** from MCP (as
`ingestion_job_recent_for_workspace`) **and** from AshAdmin.

### Red: manual `tool` duplicating a described action

```elixir
# In a domain file
tools do
  tool :answer_question, Concept.Knowledge.Tools, :answer_question do
    description "Answer a question using workspace content with citations."
  end
end
```

This shadows AutoTools' synthesis (`tools_answer_question`). The
SynthesizeTools transformer emits a compile warning. Drop the manual
entry; the description on the action itself does the work.

### Red: per-block-type custom MCP wiring

```elixir
# A new block type
defmodule Concept.Pages.BlockTypes.NewThing do
  # ... wires up handle_event, JS hook, AND a custom mcp_resource entry
  # in lib/concept/pages.ex by hand.
end
```

Use the `BlockType.{Static,Interactive}` macros. The macro handles all
of: LV event handler, JS hook, Ash action call, MCP tool exposure.
Manual wiring fragments the contract and is the kind of thing
PLAN-006's macros exist to prevent.

## Testing the contract

`test/integration/mcp_parity_test.exs` asserts:

1. Every described public action (less explicit opt-outs) is reachable
   as an MCP tool.
2. Every exposed tool has a non-nil description.
3. Every public argument on an exposed action has a description.
4. `docs/mcp_surface.md` exists.
5. `docs/mcp_surface.md` matches the live render output (zero drift).

Run via `mix test --include integration test/integration/mcp_parity_test.exs`.

CI invokes the relevant subset:

- `mix credo --strict` — LiveView purity.
- `mix concept.docs.mcp_surface --check` — surface doc drift.
- `mix test --exclude integration` — main suite.

Wired into the `precommit` alias.

## Auto-generated surface

The live MCP surface (every tool, every resource, with descriptions and
argument tables) is regenerated by `mix concept.docs.mcp_surface` into
`docs/mcp_surface.md`. The file is committed so PR review surfaces
additions and renames. CI fails on drift.

## Future

- **AshJsonApi / AshGraphql.** The same Ash actions can project to
  REST and GraphQL endpoints with the same authz. AshAi covers MCP;
  these would extend the principle to public APIs without writing any
  new code.
- **Real-time MCP subscriptions.** No client today; the existing
  pub_sub on Ash resources is the natural feed.
- **FUP-007**: Cmd-K becomes a workspace intent bar over the same
  exposed-tools set the MCP server serves. After PLAN-007, this is a
  small UI feature \u2014 the surface already exists.
- **FUP-032**: Expand `mcp_resources` to cover workspace tree, page
  blocks, chat history, etc.
- **FUP-033**: Per-block-type MCP UI resources (dynamic HTML rendering).
